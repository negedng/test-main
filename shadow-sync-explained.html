1. The Problem We're Solving

Situation: We have an internal repo (a repository with folders like backend/ and frontend/). External teams have their own separate repositories for backend and frontend. We need to keep both sides in sync — when they push changes, we get them, and when we push changes, they get them.

Challenge: Git doesn't natively support syncing a subdirectory of one repo with the root of another repo while preserving commit history (who wrote what, when, and why).

Why not git subtree? Git has a built-in command called git subtree that maps a subdirectory to an external repo. However, it has serious limitations for our use case:

  • Lossy history: git subtree push rewrites commits — the original author, date, and commit message are replaced with a single squashed commit or synthetic commits. We lose git blame accuracy and the ability to trace changes back to the person who made them.
  • No filtering: There's no built-in way to exclude files (like our .shadowignore). Every file in the subdirectory goes to the external repo, including AI prompts, IDE configs, and other internal-only files.
  • Merge pain: git subtree pull creates noisy merge commits and frequently produces conflicts that are difficult to resolve, especially when both sides are actively changing files.
  • No automation story: git subtree is designed as a manual, local-only command. There's no clean way to run it in CI on a schedule or trigger it automatically on push.

Solution: Shadow Sync uses intermediate "shadow branches" as a staging area. It replays commits one-by-one to preserve authorship, supports .shadowignore filtering, and uses GitHub Actions to automate the process.

2. Big Picture: How It All Fits Together

There are three copies of the code at any time:

NameWhereWhat
External External repo (e.g. github.com/org/backend) The external source of truth
Shadow shadow/backend/main branch on our origin A mirror — kept in sync automatically by CI
Origin Your working branch (main, feature/...) Our internal repo with all subdirectories
PULLING (external → us):
External Repo —[ CI sync every 15 min ]—→ Shadow Branch —[ git merge ]—→ Your Branch

PUSHING (us → external):
Your Branch —[ shadow-export + .shadowignore ]—→ Shadow Branch —[ CI forward ]—→ External Repo

Why shadow branches? They act as a "neutral zone" where the external repo's file structure (files at root level) is mapped to our internal repo's structure (files under backend/). Shadow branches live on our origin repository as branches named like shadow/backend/main.

3. Key Concepts (Git Basics You Need)

Repository (Repo)

A folder tracked by git. It remembers every change ever made to every file, who made it, and when.

Branch

A named line of development. Think of it as a bookmark pointing to the latest change. main is the default branch.

Commit

A snapshot of changes with a message, author, and timestamp. Each commit has a unique hash (like a1b2c3d).

Remote

A link to another copy of the repo. origin is your GitHub repo. We add external repos as additional remotes (e.g. backend).

Fetch & Push

Fetch = download new commits from a remote. Push = upload your commits to a remote.

Merge

Combine changes from one branch into another. If both sides edited the same line, you get a "conflict" to resolve manually.

Worktree

A second working copy of the repo that shares the same git history. Like having two checkout folders at once — useful for making changes on a different branch without disturbing your current work.

Orphan Branch

A branch with no parent commits — it starts completely fresh with no history. Shadow branches are created this way so they don't inherit our internal repo's unrelated history.

Trailer

Metadata appended to a commit message, like Shadow-synced-from: a1b2c3d. Used by this system to track which external commits have already been synced.

GitHub Actions

Automated scripts that run on GitHub's servers. Triggered by events (push, schedule, manual). Defined in .github/workflows/ YAML files.

4. Architecture & Data Flow

Flow A: Pulling External Changes (External → Us)

1
CI sync runs every 15 minutes (GitHub Actions cron). The workflow file shadow-sync.yml triggers shadow-ci-sync.ts.
2
Fetch from external remote. The script gets the URL from shadow-config.json and runs git fetch.
3
Discover branches. It lists all branches on the external remote (e.g. main, feature/auth).
4
Replay commits. For each branch, it checks which external commits we've already copied over (by looking at trailers in our history) and skips those. Then, for each new commit:
  1. Takes the changes that commit introduced (the diff)
  2. Applies those changes inside the corresponding folder (e.g. backend/ or frontend/) — so if the external commit changed src/app.ts, we change backend/src/app.ts
  3. Creates a matching commit on the shadow branch, keeping the original author name, email, date, and message — so it looks like the external developer made the change directly in our repo
5
Push shadow branches. The updated shadow/backend/main branch is pushed to origin.
6
You pull locally. Run npm run pull (or npm run pull -- -r backend) to merge external changes into your working branch. This uses shadow-pull.ts which merges the shadow branch but restores non-dir/ files so only backend/ is affected.

Warning: Do not use a plain git merge origin/shadow/backend/main — the shadow branch's tree only contains backend/ files, so a raw merge would delete all your other files (scripts, config, tests, etc.).

Flow B: Pushing Our Changes (Us → External)

1
You run npm run export -- -m "message" locally. This runs shadow-export.ts. (The -- tells npm to pass everything after it to the script, rather than treating it as npm's own flags.)
2
Create worktree & merge. A temporary worktree is created from the shadow branch. Your current branch (HEAD) is merged into it with git merge --no-commit, so we can clean the index before committing. The merge uses -X theirs to prefer your local changes on conflict.
3
Clean the index. The merge brought in all files from HEAD (other directories, config files, scripts…). The script strips the index back to only files under backend/, then removes any files matching .shadowignore patterns. Because MERGE_HEAD is still present, the eventual commit will record both parents — giving us real merge ancestry with a clean tree.
4
Commit & push. The result is a proper merge commit with two parents (shadow branch tip + your HEAD). Its tree only contains backend/ files, minus anything in .shadowignore. Pushed to origin.
5
CI forward triggers automatically. The push to shadow/** triggers shadow-forward.yml, which runs shadow-ci-forward.ts. It strips the backend/ prefix and pushes the files to the external remote.

7. GitHub Actions Workflows

shadow-sync.yml — "Pull"

Trigger: Cron every 15 minutes + manual dispatch

What it does:

  1. Checks out repo with full history (fetch-depth: 0)
  2. Installs Node.js 20 and runs npm ci
  3. Runs shadow-ci-sync.ts (reads remote URLs from shadow-config.json)

Concurrency: Group shadow-sync, won't cancel running syncs

Permissions: contents: write (needs to push shadow branches)

shadow-forward.yml — "Push"

Trigger: Push to any shadow/** branch

What it does:

  1. Checks out repo with full history and persist-credentials: false (disables the default credential helper so the PAT is used instead)
  2. Installs Node.js 20 and runs npm ci
  3. Configures git identity (Shadow Sync) and rewrites GitHub HTTPS URLs to use the PAT
  4. Runs shadow-ci-forward.ts (reads remote URLs from shadow-config.json)

Concurrency: Per-ref group (each shadow branch gets its own queue)

Permissions: contents: read (pushes to external, not origin). Requires an EXTERNAL_REPO_TOKEN secret for cross-repo push access — see PAT setup.

Both workflows require fetch-depth: 0 (full clone). Shadow sync needs full history to scan for trailers that track which commits have been synced. A shallow clone would cause it to fail.
The */15 * * * * cron schedule is a request, not a guarantee. GitHub Actions delays scheduled runs during high load, especially on free-tier repos. In practice, intervals of 30–60+ minutes are common. This is normal GitHub behavior — shadow sync doesn't depend on precise timing, it just means external changes may take longer to appear on the shadow branch.

8. Every Function Explained

shadow-common.ts — Shared Library

This is the core of the system. Every other script imports from here.

Configuration

Reads shadow-config.json from the same directory as the script. Parses it and fills in default values for any missing fields. Called once at module load time — the result is cached in the config constant.

Design decision: Defaults are hardcoded as fallbacks (e.g. maxBuffer defaults to 50 MB). This means the config file only needs the fields you want to customize.

Git Helpers

Runs a git command and returns its output. Throws an error if the command fails (non-zero exit code). Both run and runSafe share a private git() helper that prepends config overrides and sets buffer/encoding options.

Use when: The command must succeed (failure is a bug).

Runs a git command but never throws. Returns an object with the output, error text, exit code, and a boolean ok flag. The caller decides how to handle failure.

Use when: Failure is expected or you need to check the result (e.g. "does this branch exist?").

getCurrentBranch(): string [source:125]

Returns the name of the currently checked-out branch (e.g. "main"). Calls die() if you're in a "detached HEAD" state (not on any branch), because shadow sync requires a named branch to work with.

Lists all branches on a remote (e.g. ["main", "feature/auth"]). Filters out HEAD pointer entries. Used by CI sync to discover all branches to process, and by setup to show available branches on error.

Reads the .shadowignore file (if it exists) and returns an array of glob patterns. Returns an empty array if the file doesn't exist. Comments (lines starting with #) and blank lines are stripped. These patterns determine which files are excluded when exporting to the external repo.

Example patterns: CLAUDE.md, **/*.local, .cursor/

Pre-flight Checks

preflightChecks(externalRef): Warning[] [source:162]

Runs safety checks before syncing. Returns an array of warnings/errors. Checks for:

CheckLevelWhy
Shallow clone Error Shadow sync needs full history to track synced commits via trailers. A shallow clone is missing this history.
Submodules Warning Git submodules (repos inside repos) can't be synced as patches. They're skipped.

Utilities

Security check for remote/directory names. Rejects empty strings, .. (path traversal), absolute paths, and names starting with - (could be interpreted as git flags). Calls die() on failure.

Why: These names are used in git commands and file paths. A malicious name like --exec=rm -rf / could be catastrophic.

die(msg: string): never [source:279]

Prints an error message with a “✘” prefix and immediately exits the process with code 1. The never return type tells TypeScript this function never returns normally.

Adds a trailer line to a commit message (e.g. Shadow-synced-from: a1b2c3d). Uses git interpret-trailers for proper formatting. Falls back to manual string concatenation if the git command fails.

Why use git's trailer command? It handles edge cases like existing trailers, blank line separation, etc.

Shadow Branch Helpers

shadowBranchName(dir: string, branch: string): string [source:301]

Builds the canonical shadow branch name: shadow/{dir}/{branch}. For example, shadowBranchName("backend", "main") returns "shadow/backend/main".

The prefix defaults to "shadow" but can be changed via shadowBranchPrefix in shadow-config.json.

Lockfile

acquireLock(scriptDir: string, name: string): void [source:289]

Prevents two instances of the same script from running simultaneously. Creates a lock file in the temp directory containing the process ID (PID).

How it works:

  1. Check if a lock file exists
  2. If yes, check if the process that created it is still alive (using tasklist on Windows, kill -0 on Unix)
  3. If the process is dead, delete the stale lock and proceed
  4. If the process is alive, abort with an error
  5. Create lock file using wx flag (exclusive create — atomic, prevents race conditions)
  6. Register cleanup handlers for process exit, Ctrl+C (SIGINT), and kill (SIGTERM)

Why: Running two exports simultaneously would corrupt the shadow branch. The lock prevents this.

Replay Engine (The Core)

These private functions power the commit-by-commit replay. They are not exported — only replayCommits() is public.

private getCommitMeta(hash: string): CommitMeta [source:380]

Extracts all metadata from a commit: author name/email/date, committer name/email/date, full message, short description, and parent count. Uses a custom separator (---SHADOW-SEP---) in the git log format to safely split fields even when commit messages contain special characters.

Why parent count? Merge commits have 2+ parents and need special diff handling (diff against first parent only).

private diffForCommit(meta: CommitMeta): string [source:402]

Generates a patch (diff) for a single commit. Handles two cases:

  • Root commit (no parents): diffs against the empty tree
  • All other commits (including merges): diffs against the first parent

Uses --binary to handle binary files, -M to detect renames, and disables file mode changes (core.filemode=false) for cross-platform consistency.

private applyPatch(patch: string, subdir: string): boolean [source:398]

Applies a patch (diff) to a subdirectory using git apply --ignore-whitespace. Returns true on success, false on failure. Since shadow sync replays commits sequentially onto a clean state, a failed patch means something is fundamentally out of sync — not a merge conflict to resolve.

Design decision: Patches are written to a temp file (not piped via stdin) because Windows has issues with large stdin pipes. The temp file is always cleaned up in a finally block. CRLF line endings are normalized to LF before applying, since mixed line endings cause patch failures on Windows.

private commitWithMeta(meta: CommitMeta, message: string, allowEmpty?: boolean): void [source:463]

Creates a commit preserving the original author and committer information. Sets git environment variables (GIT_AUTHOR_NAME, etc.) so the commit appears as if it was made by the original author at the original time.

Why preserve authorship? So git blame and git log show who actually wrote each line, not "shadow sync bot".

private extractPatchFiles(patch: string, subdir: string): string[] [source:479]

Parses a patch to find which files were modified. Used to git add only the affected files (not the entire repo). Validates paths to prevent directory traversal attacks (.. or absolute paths).

private buildAlreadySyncedSetFor(dir: string): Set<string> [source:504]

Scans the local git history for commits that have a Shadow-synced-from trailer. Returns a set of external commit hashes that have already been synced. This is how the system avoids replaying the same commit twice.

Design decision: Uses trailers in commit messages (not a separate state file) because trailers travel with the commits — they work across clones, branches, and CI environments without any external state.

private findSeedHash(dir: string): string | null [source:520]

Finds the seed commit hash for a directory. The seed marks the starting point: "don't sync commits before this one." Set during shadow-setup.ts bootstrap.

private collectExternalCommits(externalRef, seedHash?): string[] [source:517]

Collects the list of commit hashes to replay, in chronological order (--reverse). If a seedHash is provided (set during setup), only collects commits after that point.

exported replayCommits(opts): { mirrored: number; upToDate: boolean } [source:564]

This is the heart of the system. It orchestrates the entire replay process:

  1. Build the set of already-synced commits (via buildAlreadySyncedSetFor)
  2. Find the seed hash (via findSeedHash)
  3. Collect all external commits that need replaying (via collectExternalCommits)
  4. Filter out already-synced ones
  5. For each new commit: generate patch → apply to subdirectory → stage files → commit with original metadata + sync trailer
  6. Handle empty patches (the commit exists in the external repo but produces no changes in our context) by recording an empty commit

Returns how many commits were mirrored and whether we were already up to date.

shadow-pull.ts — Local Pull Script

Purpose: Sync external changes and safely merge them into your local working branch. Triggers CI sync on GitHub first (if EXTERNAL_REPO_TOKEN is set), then merges the shadow branch so only dir/ files are affected.

How it works:

  1. Trigger CI sync — calls the GitHub API to dispatch the Shadow Sync workflow (external → shadow). Waits ~20 seconds for it to complete. Skipped if EXTERNAL_REPO_TOKEN is not set or --no-sync is passed.
  2. Fetch the latest shadow branch from origin
  3. Check if up to date — if shadow is already an ancestor of HEAD, nothing to do
  4. Merge with --no-commit — merges the shadow branch but stops before committing, so we can fix the index
  5. Restore non-dir/ files — checks out all files outside dir/ from HEAD, undoing the "deletions" from the shadow tree
  6. Commit — the merge commit has correct ancestry and only dir/ files were changed

Why not a plain git merge?

The shadow branch's tree only contains dir/ files (the export strips everything else). A plain git merge origin/shadow/frontend/main would tell git that all non-dir/ files were "deleted" on the shadow side, causing git to delete your scripts, config, tests, and everything else outside dir/.

Always use npm run pull, never raw git merge

This is critical. A raw git merge origin/shadow/{dir}/main will delete your files. The pull script exists specifically to prevent this. If you accidentally run a raw merge, restore with git checkout origin/main -- . and recommit.

shadow-export.ts — Local Export Script

Purpose: Export your local changes from an internal repo subdirectory to the shadow branch using a real git merge, filtering out files you don't want in the external repo.

Step-by-step flow:

  1. Parse arguments — message (required), remote, dir, branch, dry-run
  2. Check for uncommitted changes — refuses to run if the subdirectory has unsaved work (prevents exporting partial/broken state)
  3. Acquire lock (acquireLock) — prevents concurrent exports
  4. Parse .shadowignore (parseShadowIgnore) — load exclusion patterns
  5. Fetch from origin — get latest shadow branch state
  6. Create worktree — temporary checkout of the shadow branch
  7. Merge HEAD — git merge --no-commit --no-ff -X theirs --allow-unrelated-histories HEAD merges your working branch into the shadow branch. The --no-commit flag stops before committing so we can clean the index. -X theirs prefers your local changes on content conflicts. --allow-unrelated-histories handles the case where the shadow branch has no common ancestor with your working branch.
  8. Clean the index — remove all files not under dir/ (the merge brought in the entire repo), then remove files matching .shadowignore patterns. This is done via git rm --cached, which only affects the index, not the working tree. Because MERGE_HEAD is still set, the next commit will be a merge commit with two parents.
  9. Commit & push — the commit has two parents (shadow tip + HEAD) but its tree only contains dir/ files minus shadowignored ones. Retries on non-fast-forward (someone else pushed first).
  10. Cleanup — remove temporary worktree and branches

Why a merge instead of a snapshot?

The previous approach extracted every file from HEAD, filtered through .shadowignore, wiped the shadow branch worktree, and copied everything back in. This was effectively a full snapshot/replace on every export — it threw away merge history, diffs looked like full replacements, and the shadow branch had no ancestry link back to your working branch.

The merge-based approach creates a proper merge commit with two parents. This means:

  • Real merge history — git log --graph shows the relationship between your branch and the shadow branch
  • Incremental diffs — each export commit shows only what actually changed, not a full replacement
  • Git handles renames/deletions naturally — no need for manual rsync --delete logic

How .shadowignore stays hidden from the external repo

A naive merge would bring shadowignored files into the commit tree (even if deleted afterward, they'd exist in the merge parent's history). The export avoids this by cleaning the git index (staging area) before committing:

  1. git merge --no-commit stages the merged result but does not commit
  2. Files outside dir/ and shadowignored files are removed from the index via git rm --cached
  3. The commit is then created — its tree (the snapshot of files) never contains the ignored files

The merge commit does point to HEAD as a parent, but the external repo never receives HEAD's ref — it only gets the shadow branch. So the ignored files are never reachable from the external side.

Helper functions in shadow-export.ts:

globToRegex(pattern: string): RegExp [source:212]

Converts a glob pattern (like **/*.md) to a regular expression using a replace chain. Supports **/**/ (any path), * (any filename chars), ? (single char), and [...] (character classes). Patterns are pre-compiled once and reused across all files for efficiency.

shadow-setup.ts — Bootstrap Script

Purpose: One-time initialization for each remote. Run once when you first start syncing with an external repo.

What it does:

  1. Fetches from the external remote
  2. Runs pre-flight checks (shallow clone, submodules, etc.)
  3. Records a seed commit on your current branch with the trailer Shadow-seed: backend a1b2c3d, telling CI sync "start after this commit." Without it, the first CI sync would try to replay the entire history of the external repo from the very beginning.

The shadow branch itself is created automatically by CI sync on its next run — setup only needs to record where to start.

shadow-ci-sync.ts — CI Pull Script

Purpose: Runs in GitHub Actions. Fetches from all external remotes and replays new commits into shadow branches.

What it does:

  1. Iterates over all configured remotes
  2. Resolves the URL from shadow-config.json (or env var override)
  3. Adds/updates the git remote
  4. Fetches from the external remote
  5. Lists all branches on the remote
  6. For each branch, checks out the corresponding shadow branch and calls replayCommits()
  7. Pushes updated shadow branches to origin
  8. Returns to detached HEAD between branches to avoid conflicts

Error handling: If one branch fails, it logs the error, resets any partial state, and continues with the next branch. Only fails at the end if any syncs failed.

New branches: If a new branch appears on the external remote, CI sync automatically discovers it (via listExternalBranches) and creates a new shadow branch for it.

CI only — do not run locally

This script uses git checkout to switch between shadow branches in the working tree. In CI this is safe because the runner starts clean and is discarded afterward. Running it locally will switch your checkout away from your working branch, and any untracked or modified files that conflict with the shadow branch will cause errors or be overwritten. If you need to trigger a sync manually, use the GitHub Actions workflow_dispatch trigger (the "Run workflow" button in the Actions tab) rather than running the script directly.

shadow-ci-forward.ts — CI Push Script

Purpose: Runs in GitHub Actions when a shadow branch is pushed. Takes the content from the shadow branch and pushes it to the external remote.

What it does:

  1. Parses the branch name from GITHUB_REF_NAME (e.g. shadow/backend/main → dir=backend, branch=main)
  2. Finds the matching remote config entry
  3. Resolves the external remote URL from shadow-config.json
  4. Extracts files under {dir}/ from the shadow branch, stripping the directory prefix (e.g. backend/src/app.ts becomes src/app.ts)
  5. Creates a worktree from the external remote's branch
  6. Syncs files (adds new, removes deleted)
  7. Commits using the shadow branch's message, tagged with a Shadow-forwarded-from: {hash} trailer so CI sync recognizes it as a round-trip commit and skips it on pull-back
  8. Pushes to the external remote (requires EXTERNAL_REPO_TOKEN PAT)

Key details:

  • No filtering happens here — .shadowignore was already applied during export
  • The Shadow-forwarded-from trailer prevents round-trip duplicates: when CI sync pulls from external, it skips commits that originated from us
  • The export preserves CI dependencies on the shadow branch (.github/workflows/, package.json, shadow-ci-forward.ts, shadow-common.ts, shadow-config.json) so the forward workflow can run

9. Design Decisions & Why

Why per-commit replay instead of a single squashed diff?

Replaying commits one-by-one preserves the full history: who made each change, when, and why. A squashed diff would lose all intermediate commits and show everything as one big change by "sync bot." Per-commit replay means git blame and git log remain useful.

Why trailers instead of a database or state file?

Trailers are embedded in commit messages, so they travel with the code. They work in CI, in clones, and across branches without any external state to manage. There's no database to corrupt, no file to lose, and no sync state to get out of date. The downside is that scanning the log gets slower as history grows, but in practice the seed hash limits how far back it needs to look.

Why shadow branches instead of direct push/pull?

The external repo has files at root level (src/app.ts), but our internal repo has them in a subdirectory (backend/src/app.ts). Shadow branches act as a translation layer. They also decouple the timing: CI can sync on a schedule without affecting your working branch.

Why worktrees for export/forward?

A worktree lets us check out the shadow branch and merge into it without disturbing your current checkout. Without worktrees, the export script would have to stash your changes, switch branches, do the merge, switch back, and unstash — error-prone and disruptive. Worktrees are isolated and temporary.

Why cross-platform git config overrides?

Windows, macOS, and Linux handle line endings (\r\n vs \n), file permissions, and Unicode differently. The overrides (core.autocrlf=false, core.safecrlf=false, core.precomposeunicode=true, core.quotepath=false) ensure patches apply identically regardless of which OS creates or applies them.

Why a lockfile?

If two exports run simultaneously, they'd both modify the shadow branch and one would overwrite the other. The lockfile uses PID checking to detect stale locks (from crashed processes) and atomic file creation (wx flag) to prevent race conditions.

Why retry pushes?

If someone else pushes to the shadow branch between our fetch and push, git rejects the push as "non-fast-forward." The export script handles this by fetching again, rebasing onto the updated branch, and retrying (up to 3 times).

Why merge-based export instead of snapshot?

The export uses git merge --no-commit instead of extracting files and copying them. This creates a real merge commit with proper ancestry, produces clean incremental diffs, and lets git handle renames and deletions naturally. The index is cleaned before committing to strip out non-dir/ files and .shadowignore matches, so the commit tree is always clean. The script also checks for uncommitted changes upfront and refuses to run if any exist.

12. Setting Up Cross-Repo Push Access (PAT)

The CI forward workflow pushes commits from your internal repo to external repos (e.g. test-frontend, test-backend). GitHub Actions' default GITHUB_TOKEN can only push to the repo the workflow runs in — it cannot push to other repos. To forward changes to external repos, you need a Personal Access Token (PAT) with write access to those repos.

Step 1: Create a Fine-Grained Personal Access Token

  1. Go to GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens
  2. Click Generate new token
  3. Fill in:
    FieldValue
    Token nameSomething descriptive, e.g. shadow-sync-forward
    ExpirationYour choice (90 days, 1 year, etc.). You'll need to rotate it before it expires.
    Resource ownerYour GitHub account or organization
    Repository accessOnly select repositories → select each external repo (e.g. test-frontend, test-backend)
  4. Under Permissions → Repository permissions, set:

    On the external repos (e.g. test-frontend, test-backend):

    PermissionLevelWhy
    ContentsRead and writeNeeded to push commits to the external repo (CI forward)

    On the internal repo (e.g. test-main):

    PermissionLevelWhy
    ActionsRead and writeNeeded to trigger the sync workflow from npm run pull
    Leave everything else at "No access."
  5. Click Generate token and copy the token immediately — you won't be able to see it again.

Step 2: Add the Token as a Repository Secret

  1. Go to your internal repo (e.g. github.com/you/test-main) → Settings → Secrets and variables → Actions
  2. Click New repository secret
  3. Fill in:
    FieldValue
    NameEXTERNAL_REPO_TOKEN
    SecretPaste the PAT from Step 1
  4. Click Add secret

The forward workflow (shadow-forward.yml) is already configured to use this secret. It rewrites all https://github.com/ URLs to include the token, so git push to external remotes authenticates automatically.

Step 3: Set the Token Locally

The same token is used by npm run pull to trigger CI sync on demand. Set it as an environment variable:

# Linux/macOS (add to ~/.bashrc or ~/.zshrc)
export EXTERNAL_REPO_TOKEN=github_pat_...

# PowerShell (add to $PROFILE)
$env:EXTERNAL_REPO_TOKEN = "github_pat_..."

If not set, npm run pull still works — it just skips the sync trigger and pulls whatever the shadow branch currently has.

Step 4: Verify It Works

  1. Make a change in your internal repo under one of the synced directories (e.g. frontend/)
  2. Run npm run export to push to the shadow branch
  3. Check the Actions tab — the Shadow Forward workflow should trigger and push to the external repo
  4. Verify the change appears on the external repo
Fine-grained tokens expire. When your token expires, the forward workflow will start failing silently (the push step will error). Generate a new token and update the EXTERNAL_REPO_TOKEN secret. Consider setting a calendar reminder for a few days before expiration.

Why not use a classic PAT?

Classic PATs grant access to all repos in your account. Fine-grained tokens let you scope access to only the specific external repos that need it. This follows the principle of least privilege — if the token leaks, the blast radius is limited to those repos.

13. Security Considerations

Input validation

validateName() rejects remote/directory names that could be used for path traversal (..), absolute path injection (/etc/passwd), or git flag injection (--exec=...). This is critical because these names are interpolated into git commands and file paths.

.shadowignore

Prevents sensitive files (AI prompts, local configs, IDE settings) from leaking to external repos. Applied during export by removing matched files from the git index before committing. The resulting commit tree never contains ignored files, and since the external repo only receives the shadow branch (not your working branch), the ignored files are never reachable from the external side.

Committed content only

The export script merges committed state (HEAD), not the working directory. It also refuses to run if there are uncommitted changes in the target directory, preventing partial or broken state from being exported.
