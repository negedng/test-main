Shadow Sync

Technical reference — bi-directional git commit replay with path-prefix remapping

One-Pager Overview

Purpose

Shadow Sync keeps two git repositories in sync when one contains a subdirectory that mirrors the root of the other. Commits are replayed one-by-one — preserving authorship, timestamps, branch structure, and merge topology — while the path prefix is added or stripped as needed.

Each side receives a shadow/ branch that the team merges into their main branch when ready. Both teams keep full control over when changes land: no force-push, no rewritten history.

How the repos are related

RepoA (monorepo) RepoB (standalone) backend/src/app.ts <—— sync ——> src/app.ts backend/README.md <—— sync ——> README.md frontend/... (not synced) infra/... (not synced)

RepoA and RepoB are symmetric endpoints. Which one is "source" and which is "target" depends on the direction flag at runtime (--from a or --from b). The dir field in the config sets the path prefix per side ("backend" vs. "" for root).

The tool runs from a workspace — either one of the repos itself (the other is added as a remote) or a standalone orchestrator repo (both are added as remotes).

1. Configuration

A shadow-config.json file defines one or more pairs. Each pair connects two repo endpoints (a and b) with a stable name used in trailers and branch names. The file location can be overridden via the SHADOW_CONFIG environment variable; the default is shadow-config.json next to the tool. 

{
  "pairs": [
    {
      "name": "backend",
      "a": { "remote": "repo-a", "url": "https://github.com/org/repo-a.git", "dir": "backend" },
      "b": { "remote": "repo-b", "url": "https://github.com/org/repo-b.git", "dir": "" }
    }
  ],
  "trailers": { "replayed": "Shadow-replayed", "seed": "Shadow-seed" },
  "gitConfigOverrides": {
    "core.autocrlf": "false",
    "core.safecrlf": "false",
    "core.precomposeunicode": "true",
    "core.quotepath": "false"
  },
  "maxBuffer": 52428800,
  "shadowBranchPrefix": "shadow"
}

Key fields

FieldPurpose
namePair identifier. Used in shadow branch names (shadow/backend/main) and in the seed trailer.
remoteGit remote name. Use "origin" if the tool runs inside that repo.
urlRemote URL. Omit if the remote already exists (e.g. origin).
dirPath prefix on that side. "backend" means files live under backend/; "" means repo root.
trailersTrailer key bases. replayed becomes Shadow-replayed-{remote}; seed stays literal.
gitConfigOverridesPer-command -c key=value overrides for every git invocation.

2. Seed & Bootstrap

Before the first sync, you must record a seed. The seed is a marker commit that tells the replay engine where history begins — everything before it is skipped. 

npm run setup -- -r backend

shadow-setup.ts uses plumbing (git commit-tree) to build the seed commit as a child of the target branch's current tip and pushes it directly to the target remote (git push <target> <seedSHA>:refs/heads/<branch>). The workspace's HEAD is never touched. This means setup behaves identically in workspace mode (target is origin) and orchestrator mode (target is an external remote) — no more "seed landed on the wrong repo" footgun.

The seed commit carries a trailer:

Seed shadow-sync for backend from repo-b/main

Shadow-seed: backend 3fcc1ed4a5c0bdc92f4a7...

The trailer records the pair name and the tip hash of the source branch at that moment. During replay, findSeeds() scans the log for this trailer and uses matching hashes as boundaries in git rev-list to exclude ancestor commits.

Seed mapping: the engine maps seedTrailerHash ↔ seedCommitHash bidirectionally. When a replayed commit's first parent predates the seed, the mapping substitutes the seed commit itself, giving the shadow branch shared ancestry with the target's main branch — so plain git merge works.
Multi-seed (per branch): setup is additive. Running npm run setup -- -r backend -b client-x creates a second seed anchored on client-x. findSeeds() returns every seed for a pair; findSeedBoundaries() then picks, for each seed, whichever of its two hashes is reachable from any source ref and uses those as rev-list boundaries. This supports the case where both sides already have long-lived divergent branches at setup time — each branch gets its own anchor without replaying pre-seed history.

3. Steps of a Sync

One invocation of shadow-sync.ts runs the steps below. The same code handles both directions — "pull" (--from b) and "push" (--from a) — by swapping which endpoint is source vs. target.

  1. 1
    Resolve direction. Parse --from; determine source and target endpoints. validateName() rejects names containing .., leading -, or absolute paths (injection guard). Ensure both remotes exist (add or update URL if the config specifies one).
  2. 2
    Fetch from source. git fetch {source.remote}.
  3. 3
    Determine branches. If --branch is given: just that one. Otherwise: all branches on the source remote, excluding shadow/ branches.
  4. 4
    Preflight checks. For each branch, scan the source tip for shallow clones (error), case conflicts (error), submodules (warn), symlinks (warn), and LFS pointers (warn). Branches with errors are skipped; warnings are logged and replay continues.
  5. 5
    Replay commits. replayCommits() in shadow-common.ts runs the replay engine (see next section).
  6. 6
    Push shadow branches. For each branch, push the final replayed SHA to the target remote as shadow/{pairName}/{branch}. If the shadow branch has diverged (both sides have new commits), a reconciliation merge is created. See Merges.
  7. 7
    Report stale branches. If syncing all branches, warn about shadow branches whose source branch no longer exists. Print the git push --delete command for manual cleanup.

4. The Replay Engine

replayCommits() in shadow-common.ts is the core of the tool. It composes six named helpers, each responsible for one step:

  1. 1
    Scan replayed-commit mapping. scanReplayedMapping() walks the target's shadow branch and extracts Shadow-replayed-{remote}: {hash} trailers, building a Map<sourceHash, targetHash> of commits that are already replayed.
  2. 2
    Find all seeds. findSeeds() searches the local log (--all) for every Shadow-seed: {pairName} {hash} trailer matching the pair. Each entry yields a {seedTrailerHash, seedCommitHash} record. For each seed, findSeedBoundaries() tests both hashes against the source refs and keeps whichever is reachable — so a pair with multiple anchored branches produces multiple boundaries.
  3. 3
    Collect commits in topological order. collectSourceCommits() runs git rev-list --topo-order --reverse from each branch tip, excluding commits reachable from any seed boundary (^seed1 ^seed2 …) and (where dir is non-empty) narrowing to commits that touch the source subdirectory.
  4. 4
    Filter. filterNewCommits() drops commits already in the mapping (skipped — already replayed) and echo commits that carry the target's remote trailer (round-trip prevention — see Deduplication). For each echo, the original target-side hash is read out of the trailer and — if that commit is still reachable in the local repo — recorded in the SHA mapping as echo → original. Subsequent commits whose parent is the echo will then be wired to the original commit object directly.
  5. 5
    Seed mapping & fallback parent. seedShaMapping() records seedTrailerHash ↔ seedCommitHash in both directions for every seed. fallbackParent is set to the target's main tip — any source parent that still doesn't map after the echo pass (seed boundary, path filter, orphan merge, or an echo whose original was force-removed) falls back to this, so merge commits stay merges on the shadow side.
  6. 6
    Replay loop. runReplayLoop() walks the remaining commits in topo order: load .shadowignore from the source tree, compute the diff (see section 5), build a remapped tree (section 6), resolve parents via the SHA mapping, and create a new commit preserving original author, committer, and timestamps. Record the new SHA in the mapping for subsequent commits to reference.

5. How the Diff Works

The replay engine is diff-based, not snapshot-based. For each source commit, only files that changed are extracted and applied to the previous replayed tree. The result: each replayed commit shows exactly the files the source commit changed — no phantom additions or deletions from the surrounding tree.

Computing the diff

For a normal commit (has a parent):

git diff-tree -r {parent} {commit} -- {sourceDir}/

Output format: :oldmode newmode oldhash newhash status\tpath — every file that changed between the commit and its first parent, scoped to the source directory.

For a root commit (no parent):

git ls-tree -r {commit} -- {sourceDir}/

All files are treated as additions. The output is converted to diff-tree format with status A.

Supported status codes:

StatusMeaningAction
AAddedupdate-index --add --cacheinfo {mode},{hash},{path}
MModifiedupdate-index --add --cacheinfo {mode},{hash},{path}
DDeletedgit rm --cached -f {path}
RRenamedrm old path + update-index new path
CCopiedupdate-index new path (old stays)
TType changeRe-update-index at new mode

6. How the Tree Is Built

buildRemappedTree() constructs a new git tree for each replayed commit. It uses a temporary index file so the working directory is never touched.

  1. 1
    Load base tree into the temp index. If this commit has a mapped parent: git read-tree {parentTree} loads the parent's full tree. If this is the first commit: git read-tree --empty starts fresh.

    The base tree is the first mapped parent's tree — a full target-side tree. This is critical: the replayed commit contains all files in the target repo, not just the synced subdirectory. Without this, git merge would see every file outside the synced directory as deleted.
  2. 2
    Compute the diff as described in section 5.
  3. 3
    Remap paths and apply changes. For each changed file:
    • Strip source prefix: backend/src/app.ts → src/app.ts
    • Check .shadowignore: skip files matching any pattern
    • Add target prefix: src/app.ts → backend/src/app.ts (or leave as-is if target dir is root)
    • Update the temp index using the low-level action from the table above
  4. 4
    Write the tree. git write-tree produces a tree object from the temp index. That hash becomes the tree of the replayed commit.
Source commit (RepoB, dir=""): Replayed commit (RepoA, dir="backend"): diff: M src/app.ts base tree: full RepoA tree (from parent) A src/new.ts apply: M backend/src/app.ts D old-file.txt A backend/src/new.ts D backend/old-file.txt result: full RepoA tree with exactly 3 changes

7. How Merges Work

Merging happens at three distinct levels:

7a. Merge topology preservation (during replay)

When source history contains merge commits, the replay engine preserves them. resolveParents() maps each source parent hash to its replayed equivalent using the SHA mapping, and falls back to fallbackParent for anything that doesn't map (seed boundary, path filter, orphan merges). Duplicates are removed. The replayed merge is created with all resolved parents: 

git commit-tree {tree} -p {parent1} -p {parent2} ... -m {message}

Because all branches are processed in a single topological pass, a commit reachable from multiple branches is replayed once and reused — preserving shared ancestry rather than duplicating commits.

Cross-repo shared ancestry via echo mapping: if a merge on the source brought in an echo commit (one that originated on the target), the filter pass records the echo's original target-side hash in the SHA mapping (see section 4, step 4). When the merge is replayed, that parent resolves to the literal original commit — not a re-replay and not the target's current branch tip. The replayed merge ends up with one parent on the source side (replayed normally) and one parent that is the genuine target-side commit object, shared by SHA. Later git merge on the target finds the real common ancestor and avoids duplicating work.

7b. Shadow branch reconciliation (during push)

After replay, the tool pushes the result to shadow/{pair}/{branch} on the target remote. There are three cases:

CaseConditionAction
Fast-forward Current shadow tip is an ancestor of the replayed SHA Push directly.
Up to date Replayed SHA is an ancestor of the current tip Skip — nothing to push.
Diverged Neither is an ancestor of the other Create a merge commit with both as parents. The replayed tree wins. Push the merge.

In steady state the sync tool is the only writer of the shadow branch, so divergence shouldn't happen — but the tool doesn't trust that assumption. Three realistic triggers:

  1. Two workspaces racing to push. A dev laptop and a CI runner (or two CI runners) sync at roughly the same time. Both compute a replay and both try to push. The replayed commits can have different committer identity or committer timestamp depending on which machine ran the tool, so they get different commit SHAs even though author and tree are identical. The first push wins; the second arrives to find its SHA is neither an ancestor nor a descendant of the shadow tip.
  2. Force-push / history rewrite on the source branch. If someone rewrites source history (rebase, reset --hard, filter-branch), the old source SHAs are gone and the trailers on the existing shadow branch now point at dead commits. On the next sync, none of the new source commits are in the "already replayed" mapping, so the engine replays them fresh rooted on fallbackParent (the target main tip). The new chain has no ancestor relationship to the existing shadow tip. Covered by test-pull-force-rewrite.ts phase 1.
  3. Manual intervention on the shadow branch. Against policy, but possible — a human force-pushes or commits directly to shadow/backend/main.

In all three cases, the right answer is "the replayed tree is the truth, but preserve both histories." Failing loudly would turn every concurrent sync into a paging incident; the reconciliation merge keeps the diverged tip reachable so no work is lost, while the tree matches what the source currently says.

The reconciliation merge:

// Take the replayed tree as the truth
syncedTree = git rev-parse {replayedSHA}^{tree}

// Build a merge commit with both parents but the replayed tree
pushSHA = git commit-tree {syncedTree} -p {currentSHA} -p {replayedSHA}
    -m "Merge replayed commits into shadow/backend/main"

7c. User-facing merge (after sync)

After the tool pushes shadow branches, the target team lands them using plain git: 

git fetch origin
git merge origin/shadow/backend/main

Because shadow commits carry the full repo tree (not just the synced subdirectory), git's three-way merge works correctly — files outside the synced directory are untouched, and only genuine edit conflicts need manual resolution.

8. Trailers & Deduplication

In a bidirectional system, replayed commits would bounce back and forth forever without deduplication. Shadow Sync solves this with git trailers — metadata lines appended to commit messages.

Trailer format

Shadow-replayed-repo-b: 3fcc1ed4a5c0bdc92f4a7...

Every replayed commit gets a trailer whose key embeds the source remote name (sanitized: non-alphanumerics → hyphens, collapsed, trimmed) and whose value is the original source commit hash. Because the remote name lives in the key, multiple sources coexist cleanly and echo-detection stays direction-specific.

Two layers of protection

  1. Skip already-replayed commits. Phase 1 of the engine (scanReplayedMapping()) scans the target's shadow branch history and builds a sourceHash → targetSHA map. Commits already in the map are skipped this run.
  2. Skip — and remap — "echo" commits. When syncing from A to B, the replayed commit on B carries Shadow-replayed-repo-a: …. Later, when the user merges that shadow commit into B's main and we sync B back to A, the engine sees the A-sourced trailer. Two things happen:
    1. The echo commit itself is dropped from the new-commits list (no replay — that would create a duplicate).
    2. The hash inside the trailer points back at the original A-side commit. If that commit is still reachable in the local repo, the engine records echo → original in the SHA mapping. Any later commit (notably a merge) whose parent is the echo will then resolve to the original commit object itself — same SHA, shared by both sides.
    The trailer key the engine looks for is direction-specific:
    • Syncing A→B: add Shadow-replayed-repo-a, skip anything carrying Shadow-replayed-repo-b.
    • Syncing B→A: add Shadow-replayed-repo-b, skip anything carrying Shadow-replayed-repo-a.
RepoA commit X (sha = abc123) | v sync --from a RepoB shadow commit X' (carries Shadow-replayed-repo-a: abc123) | v merge into RepoB main RepoB main now contains X', plus a merge commit M whose parents are [B/main_tip, X'] | v sync --from b X' is dropped (echo); mapping recorded: X' -> abc123 M is replayed as M' with parents [replayed-B-tip, abc123] ^^^^^^ literal A-side commit, shared by SHA
Trailer matching is line-anchored: an identifier that happens to appear in a commit body (e.g. "see Shadow-replayed-repo-a: abc1234 for context") is not treated as a real trailer — only proper trailer lines at the end of the message count. See test-push-content.ts, phase 5.

9. .shadowignore

Each repo can control which files it sends to the other side. Place a .shadowignore file at the root of the synced content:

RepoConfig dirFile location
RepoA"backend"backend/.shadowignore
RepoB"" (root).shadowignore

Patterns work like .gitignore basics:

  • * matches within a single path segment
  • ** matches across path segments
  • Lines starting with # are comments
CLAUDE.md
.cursor/
**/*.local
secrets/

The ignore file is auto-discovered per commit from the source tree (git show {commit}:{dir}/.shadowignore). Adding or removing patterns takes effect immediately on the next sync — no config change required.

10. Preflight Checks

Before replaying, the engine scans each source branch for potential issues:

CheckCodeSeverityReason
Shallow clone SHALLOW_CLONE Error Replay needs full history to compute diffs and map parents.
Case-insensitive filename conflict CASE_CONFLICT Error Files like Readme.md and readme.md break on Windows and macOS.
Submodules SUBMODULE Warning Submodule entries are replayed as gitlinks, which may not resolve on the other side.
Symlinks SYMLINK Warning Symlink targets are not adjusted during prefix remapping.
Git LFS pointers GIT_LFS Warning LFS objects live in a separate store; the pointer is replayed, but the blob may not transfer.

Errors skip the offending branch. Warnings are printed and replay continues.

11. Programmatic API

The CLI entry point (shadow-sync.ts) also exports a programmatic API used by the test harness. You can run a sync in-process without shelling out: 

import { runSync, applyTestOverrides } from "shadow-sync/shadow-sync";

// Optional: inject a config without reading shadow-config.json from disk
applyTestOverrides({
  repoRoot: "/tmp/my-test-repo",
  pairs: [{
    name: "backend",
    a: { remote: "origin",      dir: "backend" },
    b: { remote: "repo-b", url: "...", dir: "" },
  }],
  shadowBranchPrefix: "shadow",
});

const result = runSync({ from: "b", pair: "backend" });
// → { exitCode: 0, stdout: "...", stderr: "..." }

runSync accepts { pair?, from?, branch? } and returns { exitCode, stdout, stderr }. This is how shadow-tests/harness.ts runs 16 test files against ephemeral git environments.
