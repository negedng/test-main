Shadow Sync

Technical Reference — Bi-directional git commit replay with path prefix remapping

One-Pager Overview

Purpose

Shadow Sync keeps two git repositories in sync when one contains a subdirectory that mirrors the root of the other. Commits are replayed individually — preserving authorship, timestamps, branch structure, and merge topology — while adding or stripping the path prefix as needed.

Each side receives a shadow/ branch that the team merges into their main branch when ready. This gives both teams full control over when changes land, without force-pushes or rewritten history.

How the repos are related

RepoA (monorepo) RepoB (standalone) backend/src/app.ts <—— sync ——> src/app.ts backend/README.md <—— sync ——> README.md frontend/... (not synced) infra/... (not synced)

RepoA and RepoB are symmetric endpoints. Which one is "source" and which is "target" depends on the direction flag at runtime (--from a or --from b). The dir field in the config defines the path prefix for each side ("backend" vs "" for root).

The tool runs from a workspace — either one of the repos themselves (the other is added as a remote) or a standalone orchestrator repo (both are added as remotes).

1. Configuration

A shadow-config.json file defines one or more pairs. Each pair connects two repo endpoints (a and b) with a stable name used in trailers and branch names. 

{
  "pairs": [{
    "name": "backend",
    "a": { "remote": "origin",       "dir": "backend" },
    "b": { "remote": "backend-repo", "url": "https://github.com/org/backend.git", "dir": "" }
  }],
  "shadowBranchPrefix": "shadow",
  "trailers": { "replayed": "Shadow-replayed", "seed": "Shadow-seed" },
  "gitConfigOverrides": { "diff.renameLimit": "10000" },
  "maxBuffer": 52428800
}

Key fields

FieldPurpose
namePair identifier. Used in shadow branch names (shadow/backend/main) and trailers.
remoteGit remote name. Use "origin" if the tool runs inside that repo.
urlRemote URL. Omit if the remote already exists (e.g. origin).
dirPath prefix in that repo. "backend" means files live under backend/. "" means root.

2. Seed & Bootstrap

Before the first sync, you must record a seed. The seed is a marker commit that tells the replay engine where history begins — everything before it is skipped. 

npm run setup -- -r backend --from a

This creates an empty commit with a trailer:

Shadow-seed: backend 3fcc1ed...

The trailer records the pair name and the tip hash of the source branch at that moment. During replay, findSeed() scans the log for this trailer and uses it as a boundary in git rev-list to exclude ancestor commits.

The seed also anchors parent mapping: when the first replayed commit's parents predate the seed, the engine maps them to the seed commit itself, giving the shadow branch shared ancestry with the target's main branch — essential for clean git merge.

3. Steps of a Sync

A single invocation of shadow-sync.ts runs through these steps. The same code handles both directions — "pull" (--from b) and "push" (--from a) — by swapping which endpoint is source vs target.

  1. 1
    Resolve direction. Parse --from, determine source and target endpoints. Validate names (no .., leading -, or absolute paths — prevents injection). Ensure both remotes exist (add/update URL if configured).
  2. 2
    Fetch from source. git fetch {source.remote} to get the latest refs.
  3. 3
    Determine branches. If --branch is given: sync only that branch. If source is the workspace (no URL): use current branch. If source is a remote: list all its branches (excluding shadow/ branches).
  4. 4
    Preflight checks. For each branch, scan the source ref for: shallow clones (error), submodules (warn), symlinks (warn), case conflicts (error), LFS pointers (warn). Branches that fail are skipped.
  5. 5
    Replay commits. The core engine (replayCommits() in shadow-common.ts) processes all branches in a single pass. Details in the sections below.
  6. 6
    Push shadow branches. For each branch, push the final replayed SHA to the target remote as shadow/{pairName}/{branch}. If the shadow branch has diverged (both sides have new commits), a merge commit is created to reconcile. See Merges.
  7. 7
    Report stale branches. If syncing all branches, warn about shadow branches whose source branch no longer exists. Print the git push --delete command for manual cleanup.

4. How the Diff Works

The replay engine is diff-based, not snapshot-based. For each source commit, only the files that changed are extracted and applied to the previous replayed tree. This produces clean, minimal diffs in the replayed history — if one file changed in the source commit, only one file appears in the replayed commit's diff.

Computing the diff

For a normal commit (has a parent):

git diff-tree -r {parent} {commit} -- {sourceDir}/

This lists every file that changed between the commit and its first parent, scoped to the source directory. Output format: :oldmode newmode oldhash newhash status\tpath

For a root commit (no parent):

git ls-tree -r {commit} -- {sourceDir}/

All files are treated as additions. The output is converted to diff-tree format with status A.

Supported status codes:

StatusMeaningAction
AAddedgit update-index --add --cacheinfo {mode},{hash},{path}
MModifiedgit update-index --add --cacheinfo {mode},{hash},{path}
DDeletedgit rm --cached -f {path}
RRenamedgit rm old path + update-index new path
CCopiedupdate-index new path (old stays)

5. How the Tree Is Built

The function buildRemappedTree() constructs a new git tree object for each replayed commit. It uses a temporary index file to assemble the tree without touching the working directory.

  1. 1
    Load base tree into temp index. If this commit has a mapped parent: git read-tree {parentTree} loads the parent's full tree. If this is the first commit: git read-tree --empty starts fresh.

    The base tree is the first mapped parent's tree, which is a full repo tree from the target side. This is critical: it means the replayed commit contains all files in the target repo, not just the synced subdirectory. Without this, git merge would see every file outside the synced directory as deleted.
  2. 2
    Compute the diff as described in section 4.
  3. 3
    Remap paths and apply changes. For each changed file:
    • Strip source prefix: remove {sourceDir}/ from the path (e.g. backend/src/app.ts → src/app.ts)
    • Check .shadowignore: skip files matching any ignore pattern
    • Add target prefix: prepend {targetDir}/ to the path (e.g. src/app.ts → backend/src/app.ts or just src/app.ts if target dir is root)
    • Update the temp index: add, modify, delete, or rename using low-level git commands
  4. 4
    Write the tree. git write-tree creates a tree object from the temp index. This tree hash is used as the tree for the replayed commit.
Source commit (RepoB, dir=""): Replayed commit (RepoA, dir="backend"): diff: M src/app.ts base tree: full RepoA tree (from parent) A src/new.ts apply: M backend/src/app.ts D old-file.txt A backend/src/new.ts D backend/old-file.txt result: full RepoA tree with 3 files changed

6. How Merges Work

Merging happens at three levels in the system:

6a. Merge topology preservation (during replay)

When the source history contains merge commits, the replay engine preserves their topology. The function resolveParents() maps each source parent hash to its replayed equivalent using the SHA mapping table. The replayed merge commit is created with all mapped parents: 

git commit-tree {tree} -p {parent1} -p {parent2} -m {message}

Because all branches are processed in a single pass (topological order), a commit that is reachable from multiple branches is replayed once and reused — preserving shared ancestry.

6b. Shadow branch reconciliation (during push)

After replay, the tool pushes the result to shadow/{pair}/{branch} on the target remote. Three cases:

CaseConditionAction
Fast-forward Current shadow branch tip is an ancestor of the replayed SHA Push directly. The shadow branch moves forward.
Up to date Replayed SHA is an ancestor of the current tip (target is ahead) Skip. Nothing to push.
Diverged Neither is an ancestor of the other Create a merge commit using commit-tree with both as parents. The replayed tree wins (ours strategy). Push the merge.

The diverged case happens when both sides make changes between syncs. The reconciliation merge:

// Get the replayed tree (this is the "truth" from the source)
syncedTree = git rev-parse {replayedSHA}^{tree}

// Create merge commit with both parents but the replayed tree
pushSHA = git commit-tree {syncedTree} -p {currentSHA} -p {replayedSHA}
    -m "Merge replayed commits into shadow/backend/main"

This preserves both histories as parents but resolves the content to match the source.

6c. User-facing merge (after sync)

After the tool pushes shadow branches, the target team merges them into their main branch using standard git: 

git fetch origin
git merge origin/shadow/backend/main

Because shadow commits carry the full repo tree (not just the synced subdirectory), git's three-way merge works correctly — files outside the synced directory are untouched, and only actual conflicts require manual resolution.

7. Deduplication & Round-Trip Prevention

In a bidirectional system, replayed commits would bounce back and forth forever without deduplication. Shadow Sync solves this with git trailers — metadata lines appended to commit messages.

Trailer format

Shadow-replayed (repo-b): 3fcc1ed4a...

Every replayed commit gets a trailer recording the source remote name and the original commit hash.

Two layers of protection

  1. Skip already-replayed commits. Before replay, buildTrailerMapping() scans the target's shadow branch history and builds a hash → SHA map of every commit that's already been replayed. These are skipped.
  2. Skip "echo" commits. When syncing from A to B, the replayed commit on B carries Shadow-replayed (repo-a): .... Later, when syncing B back to A, the engine sees this trailer and knows the commit originated from A — it skips replaying it back. The trailer key to look for is direction-specific:
    • Syncing A→B: add Shadow-replayed (repo-a), skip commits containing Shadow-replayed (repo-b)
    • Syncing B→A: add Shadow-replayed (repo-b), skip commits containing Shadow-replayed (repo-a)
RepoA commit: "Add feature X" | v sync --from a RepoB shadow commit: "Add feature X Shadow-replayed (repo-a): abc123" | v merge into RepoB main RepoB commit: "Add feature X Shadow-replayed (repo-a): abc123" | v sync --from b SKIPPED — trailer says this came from repo-a

8. .shadowignore

Each repo can control which files it sends to the other side. Place a .shadowignore file at the root of the synced content:

RepoConfig dirFile location
RepoA"backend"backend/.shadowignore
RepoB"" (root).shadowignore

Patterns work like .gitignore basics:

  • * matches within a single path segment
  • ** matches across path segments
  • Lines starting with # are comments
CLAUDE.md
.cursor/
**/*.local
secrets/

The ignore file is auto-discovered per commit from the source tree (git show {commit}:{path}/.shadowignore), so adding or removing patterns takes effect immediately on the next sync without any configuration change.

9. Preflight Checks

Before replaying, the engine scans each source branch for potential issues:

CheckSeverityReason
Shallow clone Error Replay needs full history to compute diffs and map parents.
Case-insensitive filename conflict Error Files like Readme.md and readme.md break on Windows/macOS.
Submodules Warning Submodule entries are replayed as gitlinks, which may not resolve on the other side.
Symlinks Warning Symlink targets are not adjusted during prefix remapping.
Git LFS pointers Warning LFS objects exist in a separate store; the pointer is replayed but the blob may not transfer.

Branches with errors are skipped. Warnings are printed but replay continues.
