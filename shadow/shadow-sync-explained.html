1. The Problem

Challenge Two repos, one codebase subset

A common pattern in organizations: RepoA is a monorepo with many directories, one of which is backend/. RepoB is a standalone repository containing just the backend code at its root. Both repos are actively developed. Changes made in one need to appear in the other, with the path prefix (backend/) added or stripped accordingly.

Manual cherry-picking breaks down quickly. Subtree merges lose authorship. Git submodules add friction and cannot remap paths. What is needed is an automated, bi-directional sync that:

  • Replays commits per-commit (preserving git blame and authorship)
  • Adds or strips the path prefix as needed
  • Preserves merge topology and branch structure
  • Prevents infinite round-trip duplication
  • Works from either repo or from a standalone orchestrator

2. Architecture

Core One script, two directions

Shadow Sync is a single script (shadow-sync.ts) driven by a --from flag that sets the direction. The shared replay engine in shadow-common.ts handles both directions with identical logic. 

# Pull: replay RepoB's commits into shadow branches on RepoA
shadow-sync --from b

# Push: replay RepoA's commits into shadow branches on RepoB
shadow-sync --from a

Shadow branches are created under a configurable prefix. The default naming convention is shadow/{pairName}/{branch}, for example shadow/backend/main.

Mode Workspace mode

The tool runs from inside one of the two repos. That repo is "origin" (no URL needed in config). The other repo is added as a remote. This is the typical setup for a developer or CI environment.

The tool runs from a separate, standalone git repository. Both endpoints have URLs configured. Neither repo is the "home" repo — the orchestrator is a neutral third party.

4. How Replay Works

Engine Unified replay: replayCommits()

The core function replayCommits({ pair, from: "a"|"b", branches }) drives both directions with a single code path. The direction flag selects which endpoint is the source and which is the target, and which trailers to add versus skip.

Replay steps (per commit)

1
Read source tree. If the source endpoint has a non-empty dir (e.g. "backend"), use git rev-parse commit:backend to extract just that subtree. If dir is empty, use the full tree of the commit.
2
Place under target prefix. If the target endpoint has a non-empty dir, use git read-tree --prefix=backend/ to nest the tree under the target directory. If dir is empty, use the tree as-is.
3
Overlay onto graft base. If the target has a dir prefix, overlay the remapped subtree onto a base tree containing the full repo. This ensures shadow commits carry the complete repo tree so git merge does not delete files outside the synced directory.
4
Apply .shadowignore. If the source commit contains a .shadowignore file, remove matching files from the temporary index before writing the tree.
5
Write tree and commit. Run git write-tree to finalize the tree object, then git commit-tree with the original author name, email, date, committer name, email, date, and the commit message with the appropriate direction-specific trailer appended.
6
Record SHA mapping. Store source SHA → replayed SHA so that parent resolution and deduplication work correctly for subsequent commits in the same batch.

Function buildRemappedTree()

A unified tree builder used by both directions. It handles:

  • Extracting a subdirectory tree from a commit (git rev-parse {hash}:{dir})
  • Adding a prefix to a tree (git read-tree --prefix={dir}/)
  • Overlaying onto a base tree (read base tree first, then overlay the remapped content)
  • Applying .shadowignore by removing matched paths from the temp index
  • Producing a final tree hash via git write-tree

Topology preservation

All branches are processed in a single topological pass. When a commit is an ancestor shared by multiple branches, it is replayed once and reused via the SHA mapping. Merge commits have their parents mapped through the same mapping table, preserving the original branch and merge topology.

The function resolveParents(commit, shaMapping, graftBase) handles parent mapping. If a parent has not been replayed (e.g. it predates the seed), the graft base is used as a fallback parent.

5. Trailers & Deduplication

Critical Preventing round-trip duplication

Without safeguards, syncing A→B→A would create infinite duplicates. Shadow Sync uses git commit trailers (footer lines in the commit message) as the deduplication mechanism. Two complementary trailer types work together:

B → A Shadow-replayed: {b.remote}

Added to every commit replayed from B into A. Records the original commit hash from RepoB. 

Shadow-replayed: repo-b a1b2c3d

When syncing A→B, any commit carrying this trailer is skipped — it originated in B and should not be sent back.

A → B Shadow-replayed: {a.remote}

Added to every commit replayed from A into B. Records the original commit hash from RepoA. 

Shadow-replayed: repo-a e4f5g6h

When syncing B→A, any commit carrying this trailer is skipped — it originated in A and should not be sent back.

Direction config (inside replayCommits)

DirectionaddTrailerskipTrailer
A → B (--from a)Shadow-replayed: {a.remote}Shadow-replayed: {b.remote}
B → A (--from b)Shadow-replayed: {b.remote}Shadow-replayed: {a.remote}

Seed Shadow-seed trailer

When shadow-setup.ts bootstraps the sync, it creates a seed commit with the trailer: 

Shadow-seed: backend a1b2c3d

This records the starting point. During replay, all commits before the seed boundary are skipped. The seed hash is found by findSeedHash(pairName), and the workspace commit that recorded the seed is found by findSeedCommit(pairName).

Used in CI workflows. The shadow-forward.yml workflow only processes pushes to shadow branches Both directions run on independent cron schedules. This prevents cascade loops where triggering on every shadow branch update and limits it to intentional exports.

6. Graft Base (Full-Repo Tree)

Important Why shadow commits need the full tree

When the target endpoint has a non-empty dir (e.g. backend/), shadow branches will contain commits that only have content under backend/. If you merge such a branch into the target repo's main branch, git sees all files outside backend/ as deleted and removes them.

The graft base solves this: each shadow commit is built on top of a base tree that contains the full repo. The synced subtree is overlaid onto this base, producing a complete tree. This way, git merge works correctly because both sides have the same set of top-level directories.

How the graft base is selected

ScenarioGraft base source
Workspace is the target repo The seed commit (the workspace commit where Shadow-seed: was recorded)
Running from an orchestrator The target repo's main branch tip
Target dir is "" (root) No graft base needed — the synced content IS the entire tree

7. .shadowignore

Filter Per-endpoint file exclusion

Works like .gitignore — commit a .shadowignore file in your repo and it is automatically discovered from the source commit's tree during replay. This file contains glob patterns (one per line) for files that this endpoint should not send to the other side. 

# .shadowignore example
# CI configs specific to this repo
.github/
.gitlab-ci.yml

# Local tooling
.vscode/
.idea/

# Generated files
dist/
coverage/

The ignore filtering is applied during buildRemappedTree(). After the tree is constructed in the temporary index, any files matching the ignore patterns are removed from the index before git write-tree finalizes the tree hash.

Note: Place .shadowignore at the root of the synced content (e.g. backend/.shadowignore in RepoA, .shadowignore in RepoB). Each side controls what RepoA sends to RepoB (during --from a). RepoB's ignore controls what RepoB sends to RepoA (during --from b).

11. Design Decisions

1
Per-commit replay preserves authorship. Each commit is replayed individually with the original author and committer metadata. This means git blame, git log --author, and contribution statistics work correctly on both sides.
2
Full-repo tree in shadow commits enables plain git merge. By grafting onto a base tree, shadow branches carry the complete repository tree. Merging a shadow branch into the target repo's main branch is a standard merge operation with no special flags or tricks needed.
3
Topology-preserving. All branches are processed in one topological pass. Shared ancestors are replayed once and reused. Merge commits retain their merge structure. The resulting shadow history is a faithful mirror of the source's branch and merge topology.
4
Trailers instead of state files. Deduplication state is embedded in the commit messages themselves, not in external files or databases. This makes the system portable across clones, CI environments, and orchestrators — any clone of the repo carries the full sync state.
5
Git plumbing commands: fast, no worktree needed. The replay engine uses git read-tree, git write-tree, and git commit-tree instead of checkout-based workflows. This means no files are written to disk, no worktree is needed, and the process is fast even for large repositories.
6
Cross-platform git config overrides. The gitConfigOverrides field in config applies settings like core.autocrlf=false to every git command, ensuring consistent behavior across Windows, macOS, and Linux.
7
Symmetric a/b config. Endpoints are labeled a and b with no inherent hierarchy. The same tool and config works from either side. Direction is purely a runtime choice.
8
Orchestrator support. When running from a third-party repo, the graft base falls back to the target's main branch tip instead of requiring a local seed commit. This enables centralized CI workflows.

13. GitHub Actions

Pull shadow-sync.yml

Trigger: Cron schedule (every 15 minutes, disabled until real URLs are configured).

Action: Runs shadow-sync.ts --from b to pull new commits from the external repo into shadow branches.

Flow:

  1. Checkout the repository
  2. Install dependencies
  3. Run shadow-sync --from b
  4. Push updated shadow branches

Push shadow-forward.yml

Trigger: Push to shadow/** branches.

Guard: Only processes commits that contain the Shadow-replayed ( trailer. This prevents the workflow from triggering on every shadow branch update.

Action: Runs shadow-sync.ts --from a to push commits to the external repo.

Secret: Requires EXTERNAL_REPO_TOKEN (a fine-grained PAT) to authenticate with the external repository.

14. PAT Setup (Fine-Grained GitHub Token)

Auth Creating a fine-grained Personal Access Token

The push workflow (shadow-forward.yml) needs a GitHub token to push to the external repository. A fine-grained PAT scoped to just the target repository is the recommended approach.

1
Navigate to token settings. Go to GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens. Click Generate new token.
2
Configure token scope.
  • Token name: Something descriptive, e.g. shadow-sync-push-repoB
  • Expiration: Choose based on your security policy (90 days is common)
  • Resource owner: Select the organization or user that owns the target repo
  • Repository access: Select Only select repositories and choose the target repo
3
Set permissions. Under Repository permissions, grant:
  • Contents: Read and write (needed to push commits and branches)
  • Metadata: Read-only (automatically selected)
All other permissions should remain at No access.
4
Generate and save the token. Click Generate token. Copy the token immediately — it will not be shown again.
5
Add as a repository secret. In the source repository (where the workflow runs), go to Settings → Secrets and variables → Actions. Click New repository secret. Name it EXTERNAL_REPO_TOKEN and paste the token value.
Security note: Fine-grained tokens are scoped to specific repositories and have explicit permissions, making them significantly more secure than classic PATs. Always use the minimum permissions required. Set an expiration date and rotate tokens before they expire.

15. Git Commands Reference

Plumbing Core git plumbing used by Shadow Sync

Shadow Sync operates entirely via git plumbing commands, avoiding porcelain commands that require a working tree. Here are the key commands and how they are used:

git read-tree --prefix={dir}/ Place a tree under a directory prefix in the index. Used when the target has a non-empty dir.
git read-tree Load a tree into the index. Used to set up the graft base tree before overlaying synced content.
git write-tree Write the current index contents as a tree object. Produces the final tree hash for each replayed commit.
git commit-tree Create a commit object from a tree hash, parent hash(es), and metadata. The core commit creation step.
git update-ref Point a branch ref to a specific commit. Used to update shadow branch pointers after replay.
git cat-file Read git object contents. Used in getCommitMeta() to extract author/committer/message fields.
git log --format=... Walk commit history with custom format strings. Used for trailer scanning, topo ordering, and seed discovery.

16. Security

Input validation validateName()

All user-supplied names (pair names, remote names, branch names) pass through validateName(), which rejects:

  • Path traversal: Values containing .. are rejected to prevent directory escape.
  • Absolute paths: Values starting with / or containing : (Windows drive letters) are rejected.
  • Flag injection: Values starting with - are rejected to prevent them from being interpreted as command-line flags by git.

Credential handling Token security

  • Tokens are stored as GitHub repository secrets, never in code or config files.
  • Fine-grained PATs are scoped to the minimum required repository and permissions.
  • The shadow-config.json file contains only remote URLs (which are public for public repos) — never tokens or credentials.
  • In CI, the token is injected via the URL: https://x-access-token:{token}@github.com/org/repo.git. This is handled by the workflow, not by the sync tool itself.

Preflight safety checks

Before replaying commits, preflightChecks() scans the source for potential issues:

CheckSeverityReason
Shallow clone Error Shallow clones lack full history. Replaying from a shallow clone would produce incomplete or incorrect results.
Submodules Warning Submodule references (gitlinks) cannot be meaningfully replayed. The file will appear as a broken reference in the target.
Symlinks Warning Symlinks may point to paths that do not exist after prefix remapping, leading to broken links.
Case conflicts Warning Files differing only in case (e.g. README.md vs readme.md) cause problems on case-insensitive filesystems (Windows, macOS default).
LFS objects Warning Git LFS pointer files will be replayed as-is. The target repo needs LFS configured to resolve them.

No working tree exposure

Shadow Sync operates entirely on git objects (trees, blobs, commits) via plumbing commands. No files are written to disk during replay, which means:

  • No risk of overwriting uncommitted work in the working directory.
  • No temporary files that could leak sensitive content.
  • The process can run safely in CI without a clean checkout.
