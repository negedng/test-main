1. The Problem We're Solving

Situation: We have an internal repo (a repository with folders like backend/ and frontend/). External teams have their own separate repositories for backend and frontend. We need to keep both sides in sync — when they push changes, we get them, and when we push changes, they get them.

Challenge: Git doesn't natively support syncing a subdirectory of one repo with the root of another repo while preserving commit history (who wrote what, when, and why).

Why not git subtree? Git has a built-in command called git subtree that maps a subdirectory to an external repo. However, it has serious limitations for our use case:

  • Lossy history: git subtree push rewrites commits — the original author, date, and commit message are replaced with a single squashed commit or synthetic commits. We lose git blame accuracy and the ability to trace changes back to the person who made them.
  • No filtering: There's no built-in way to exclude files (like our .shadowignore). Every file in the subdirectory goes to the external repo, including AI prompts, IDE configs, and other internal-only files.
  • Merge pain: git subtree pull creates noisy merge commits and frequently produces conflicts that are difficult to resolve, especially when both sides are actively changing files.
  • No automation story: git subtree is designed as a manual, local-only command. There's no clean way to run it in CI on a schedule or trigger it automatically on push.

Solution: Shadow Sync uses intermediate "shadow branches" as a staging area. It replays commits one-by-one to preserve authorship, supports .shadowignore filtering, and uses GitHub Actions to automate the process.

2. Big Picture: How It All Fits Together

There are three copies of the code at any time:

NameWhereWhat
External External repo (e.g. github.com/org/backend) The external source of truth
Shadow shadow/backend/main branch on our origin A mirror — kept in sync automatically by CI
Origin Your working branch (main, feature/...) Our internal repo with all subdirectories
PULLING (external → us):
External Repo —[ CI sync every 15 min ]—→ Shadow Branch —[ git merge ]—→ Your Branch

PUSHING (us → external):
Your Branch —[ shadow-export + .shadowignore ]—→ Shadow Branch —[ CI forward ]—→ External Repo

Why shadow branches? They act as a "neutral zone" where the external repo's file structure (files at root level) is mapped to our internal repo's structure (files under backend/). Shadow branches live on our origin repository as branches named like shadow/backend/main.

Here's a real example of one full cycle: a local developer adds server.ts, an external developer adds routes.ts, and the local developer adds middleware.ts.

Shadow branch file tree (shadow/backend/main)

The shadow branch contains the backend/ files plus CI dependencies needed for the forward workflow to run:

shadow/backend/main
├── .github/workflows/
│   ├── shadow-forward.yml          ← CI forward workflow (must be on the branch to trigger)
│   └── shadow-sync.yml
├── backend/
│   ├── middleware.ts                ← your files live under backend/
│   ├── routes.ts
│   ├── README.md
│   └── server.ts
└── shadow/
    ├── package.json                 ← needed for npm ci in the forward workflow
    ├── package-lock.json
    ├── shadow-ci-forward.ts         ← the forward script itself
    ├── shadow-common.ts             ← shared helpers
    └── shadow-config.json           ← remote/dir config

Everything else from your internal repo (other subdirectories, test files, export scripts, docs) is stripped during export and never appears on the shadow branch.

Commit history on the internal repo (main)

* Export backend/ (2 commits)                      ← export: pushes database.ts + logger.ts to shadow
|   - Add logger utility                           ← auto-generated message lists each commit
|   - Add database config
* Add logger utility                               ← your commit
* Add database config                              ← your commit
* Merge remote-tracking branch 'origin/shadow/...' ← pull: brings routes.ts from shadow
* Add API routes                                   ← synced from external (preserves author)
|   Shadow-synced-from: ad827ff                    ← trailer: tracks which external commit this was
* Export backend/ (1 commit)                       ← export: pushes server.ts to shadow
|   - Add backend API server
* Add backend API server                           ← your commit

Each change creates 2 commits: the real commit + a merge commit (either from pull or export). The merge commits record ancestry between your branch and the shadow branch. Export messages auto-summarize the included commits; use -m to override.

Commit history on the external repo (test-backend)

* Export backend/ (2 commits)               ← forwarded from shadow (database.ts + logger.ts)
|   - Add logger utility
|   - Add database config
|   Shadow-forwarded-from: 2435080          ← trailer: CI sync will skip this on pull-back
* Add API routes                             ← external developer's commit
* Export backend/ (1 commit)                 ← forwarded from shadow (server.ts)
|   - Add backend API server
|   Shadow-forwarded-from: c4c625c          ← trailer: prevents round-trip duplicate
* Initial commit

Forwarded commits have a single Shadow-forwarded-from trailer. When CI sync sees this trailer, it skips the commit instead of trying to replay it back (which would fail since the content already exists on the shadow branch).

In steady-state usage, each real change produces exactly 2 commits on each side: the content commit + the export/forward merge commit.

3. Key Concepts (Git Basics You Need)

Repository (Repo)

A folder tracked by git. It remembers every change ever made to every file, who made it, and when.

Branch

A named line of development. Think of it as a bookmark pointing to the latest change. main is the default branch.

Commit

A snapshot of changes with a message, author, and timestamp. Each commit has a unique hash (like a1b2c3d).

Remote

A link to another copy of the repo. origin is your GitHub repo. We add external repos as additional remotes (e.g. backend).

Fetch & Push

Fetch = download new commits from a remote. Push = upload your commits to a remote.

Merge

Combine changes from one branch into another. If both sides edited the same line, you get a "conflict" to resolve manually.

Worktree

A second working copy of the repo that shares the same git history. Like having two checkout folders at once — useful for making changes on a different branch without disturbing your current work. In Shadow Sync, only ci-sync uses worktrees (for replaying commits). Export and ci-forward use git plumbing with a temp index instead.

Orphan Branch

A branch with no parent commits — it starts completely fresh with no history. Shadow branches are created this way so they don't inherit our internal repo's unrelated history.

Trailer

Metadata appended to a commit message, like Shadow-synced-from: a1b2c3d. Used by this system to track which external commits have already been synced.

GitHub Actions

Automated scripts that run on GitHub's servers. Triggered by events (push, schedule, manual). Defined in .github/workflows/ YAML files.

4. Architecture & Data Flow

Flow A: Pulling External Changes (External → Us)

1
CI sync runs every 15 minutes (GitHub Actions cron). The workflow file shadow-sync.yml triggers shadow-ci-sync.ts.
2
Fetch from external remote. The script gets the URL from shadow-config.json and runs git fetch.
3
Discover branches. It lists all branches on the external remote (e.g. main, feature/auth).
4
Replay commits. For each branch, it checks which external commits we've already copied over (by looking at trailers in our history) and skips those. Then, for each new commit:
  1. Takes the changes that commit introduced (the diff)
  2. Applies those changes inside the corresponding folder (e.g. backend/ or frontend/) — so if the external commit changed src/app.ts, we change backend/src/app.ts
  3. Creates a matching commit on the shadow branch, keeping the original author name, email, date, and message — so it looks like the external developer made the change directly in our repo
5
Push shadow branches. The updated shadow/backend/main branch is pushed to origin.
6
You import locally. Run npm run import (or npm run import -- -r backend) to merge external changes into your working branch. This uses shadow-import.ts which merges the shadow branch but restores non-dir/ files so only backend/ is affected.

Warning: Do not use a plain git merge origin/shadow/backend/main — the shadow branch's tree only contains backend/ files, so a raw merge would delete all your other files (scripts, config, tests, etc.).

Flow B: Pushing Our Changes (Us → External)

1
You run npm run export -- -m "message" locally. This runs shadow-export.ts. (The -- tells npm to pass everything after it to the script, rather than treating it as npm's own flags.)
2
Build tree using temp index. Instead of a worktree or merge, the script uses git plumbing commands with a temporary index file (set via GIT_INDEX_FILE). It runs git read-tree --empty to start clean, then git read-tree --prefix=dir/ HEAD:dir to read only the wanted subtrees into the index. It also reads .github/ and shadow/ for CI dependencies.
3
Filter and write tree. Files matching .shadowignore patterns are removed from the index via git rm --cached. Then git write-tree creates a tree object from the index. The tree is compared against the shadow branch's tree (git rev-parse shadowRef^{tree}) to check whether there are any actual changes to export.
4
Create merge commit and push. Uses git commit-tree with two parents (shadow branch tip + HEAD) to create a proper merge commit without needing a working directory. The commit is pushed directly to origin. No retry loop — if the push fails (non-fast-forward), the user must merge the shadow branch and retry.
5
CI forward triggers automatically. The push to shadow/** triggers shadow-forward.yml, which checks for the Shadow-export: trailer (only export commits are forwarded, not ci-sync mirrors). It runs shadow-ci-forward.ts, which strips the backend/ prefix and pushes the files to the external remote.

7. GitHub Actions Workflows

shadow-sync.yml — "Pull"

Trigger: Cron every 15 minutes + manual dispatch

What it does:

  1. Checks out repo with full history (fetch-depth: 0)
  2. Installs Node.js 20 and runs npm ci
  3. Runs shadow-ci-sync.ts (reads remote URLs from shadow-config.json)

Concurrency: Group shadow-sync, won't cancel running syncs

Permissions: contents: write (needs to push shadow branches)

shadow-forward.yml — "Push"

Trigger: Push to any shadow/** branch, but only for export commits (checks for Shadow-export: trailer in the commit message — ci-sync commits are skipped)

What it does:

  1. Checks out repo with full history and persist-credentials: false (disables the default credential helper so the PAT is used instead)
  2. Installs Node.js 20 and runs npm ci
  3. Configures git identity (Shadow Sync) and rewrites GitHub HTTPS URLs to use the PAT
  4. Runs shadow-ci-forward.ts (reads remote URLs from shadow-config.json)

Concurrency: Per-ref group (each shadow branch gets its own queue)

Permissions: contents: read (pushes to external, not origin). Requires an EXTERNAL_REPO_TOKEN secret for cross-repo push access — see PAT setup.

Both workflows require fetch-depth: 0 (full clone). Shadow sync needs full history to scan for trailers that track which commits have been synced. A shallow clone would cause it to fail.
The */15 * * * * cron schedule is a request, not a guarantee. GitHub Actions delays scheduled runs during high load, especially on free-tier repos. In practice, intervals of 30–60+ minutes are common. This is normal GitHub behavior — shadow sync doesn't depend on precise timing, it just means external changes may take longer to appear on the shadow branch.

8. Every Function Explained

shadow-common.ts — Shared Library

This is the core of the system. Every other script imports from here.

Configuration

loadConfig(): ShadowSyncConfig [source:30]

Reads shadow-config.json from the same directory as the script. Parses it and fills in default values for any missing fields. Called once at module load time — the result is cached in the config constant.

Design decision: Defaults are hardcoded as fallbacks (e.g. maxBuffer defaults to 50 MB). This means the config file only needs the fields you want to customize.

Git Helpers

git(args, opts?): string | GitResult [source:93]

Single entry point for all git commands. Options:

  • { safe: true } — returns { stdout, stderr, status, ok } instead of throwing. Use when failure is expected (e.g. checking if a ref exists).
  • { plain: true } — skips config overrides (core.autocrlf=false, etc.). Use for working-tree operations on Windows where the repo's own settings must be respected.
  • Both can be combined: { safe: true, plain: true }

Without safe, throws on non-zero exit. Without plain, prepends cross-OS config overrides for consistent replay behavior. All calls default to the repo root as cwd (detected via git rev-parse --show-toplevel).

getCurrentBranch(): string [source:108]

Returns the name of the currently checked-out branch (e.g. "main"). Calls die() if you're in a "detached HEAD" state (not on any branch), because shadow sync requires a named branch to work with.

refExists(ref: string): boolean [source:104]

Checks if a git reference (branch, tag, commit) exists. Uses git rev-parse --verify. Returns true or false.

Example: refExists("origin/shadow/backend/main") → does this shadow branch exist on origin?

listExternalBranches(remote: string): string[] [source:114]

Lists all branches on a remote (e.g. ["main", "feature/auth"]). Filters out HEAD pointer entries. Used by CI sync to discover all branches to process, and by setup to show available branches on error.

.shadowignore

parseShadowIgnore(scriptDir: string): string[] [source:140]

Reads the .shadowignore file (if it exists) and returns an array of glob patterns. Returns an empty array if the file doesn't exist. Comments (lines starting with #) and blank lines are stripped. These patterns determine which files are excluded when exporting to the external repo.

Example patterns: CLAUDE.md, **/*.local, .cursor/

Pre-flight Checks

preflightChecks(externalRef): Warning[] [source:220]

Runs safety checks before syncing. Returns an array of warnings/errors. Checks for:

CheckLevelWhy
Shallow clone Error Shadow sync needs full history to track synced commits via trailers. A shallow clone is missing this history.
Submodules Warning Git submodules (repos inside repos) can't be synced as patches. They're skipped.
Symlinks Warning Symlink targets might point to wrong paths after being moved into a subdirectory.
Case conflicts Error Files like Readme.md and README.md collide on Windows/macOS (case-insensitive file systems). This causes data loss.
Git LFS Warning If the remote uses Large File Storage, synced files will be pointer files unless LFS is also configured locally.
handlePreflightResults(warnings): boolean [source:253]

Prints all warnings/errors to the console. Returns false if any errors were found (caller should abort). Warnings are printed but don't block execution.

Utilities

validateName(value: string, label: string): void [source:76]

Security check for remote/directory names. Rejects empty strings, .. (path traversal), absolute paths, and names starting with - (could be interpreted as git flags). Calls die() on failure.

Why: These names are used in git commands and file paths. A malicious name like --exec=rm -rf / could be catastrophic.

die(msg: string): never [source:70]

Prints an error message with a “✘” prefix and immediately exits the process with code 1. The never return type tells TypeScript this function never returns normally.

appendTrailer(message: string, trailer: string): string [source:124]

Adds a trailer line to a commit message (e.g. Shadow-synced-from: a1b2c3d). Uses git interpret-trailers for proper formatting. Falls back to manual string concatenation if the git command fails.

Why use git's trailer command? It handles edge cases like existing trailers, blank line separation, etc.

Shadow Branch Helpers

shadowBranchName(dir: string, branch: string): string [source:121]

Builds the canonical shadow branch name: shadow/{dir}/{branch}. For example, shadowBranchName("backend", "main") returns "shadow/backend/main".

The prefix defaults to "shadow" but can be changed via shadowBranchPrefix in shadow-config.json.

Lockfile

acquireLock(scriptDir: string, name: string): void [source:148]

Prevents two instances of the same script from running simultaneously. Creates a lock file in the temp directory containing the process ID (PID).

How it works:

  1. Check if a lock file exists
  2. If yes, check if the process that created it is still alive (using tasklist on Windows, kill -0 on Unix)
  3. If the process is dead, delete the stale lock and proceed
  4. If the process is alive, abort with an error
  5. Create lock file using wx flag (exclusive create — atomic, prevents race conditions)
  6. Register cleanup handlers for process exit, Ctrl+C (SIGINT), and kill (SIGTERM)

Why: Running two exports simultaneously would corrupt the shadow branch. The lock prevents this.

Replay Engine (The Core)

These private functions power the commit-by-commit replay. They are not exported — only replayCommits() is public.

private getCommitMeta(hash: string): CommitMeta [source:278]

Extracts all metadata from a commit: author name/email/date, committer name/email/date, full message, short description, and parent count. Uses a custom separator (---SHADOW-SEP---) in the git log format to safely split fields even when commit messages contain special characters.

Why parent count? Merge commits have 2+ parents and need special diff handling (diff against first parent only).

private diffForCommit(meta: CommitMeta): string [source:300]

Generates a patch (diff) for a single commit. Handles two cases:

  • Root commit (no parents): diffs against the empty tree
  • All other commits (including merges): diffs against the first parent

Uses --binary to handle binary files, -M to detect renames, and disables file mode changes (core.filemode=false) for cross-platform consistency.

private applyPatch(patch: string, subdir: string): boolean [source:317]

Applies a patch (diff) to a subdirectory using git apply --ignore-whitespace. Returns true on success, false on failure. Since shadow sync replays commits sequentially onto a clean state, a failed patch means something is fundamentally out of sync — not a merge conflict to resolve.

Design decision: Patches are written to a temp file (not piped via stdin) because Windows has issues with large stdin pipes. The temp file is always cleaned up in a finally block. CRLF line endings are normalized to LF before applying, since mixed line endings cause patch failures on Windows.

private commitWithMeta(meta: CommitMeta, message: string, allowEmpty?: boolean): void [source:347]

Creates a commit preserving the original author and committer information. Sets git environment variables (GIT_AUTHOR_NAME, etc.) so the commit appears as if it was made by the original author at the original time.

Why preserve authorship? So git blame and git log show who actually wrote each line, not "shadow sync bot".

private extractPatchFiles(patch: string, subdir: string): string[] [source:363]

Parses a patch to find which files were modified. Used to git add only the affected files (not the entire repo). Validates paths to prevent directory traversal attacks (.. or absolute paths).

private buildAlreadySyncedSetFor(dir: string): Set<string> [source:376]

Scans the local git history for commits that have a Shadow-synced-from trailer. Returns a set of external commit hashes that have already been synced. This is how the system avoids replaying the same commit twice.

Design decision: Uses trailers in commit messages (not a separate state file) because trailers travel with the commits — they work across clones, branches, and CI environments without any external state.

private findSeedHash(dir: string): string | null [source:392]

Finds the seed commit hash for a directory. The seed marks the starting point: "don't sync commits before this one." Set during shadow-setup.ts bootstrap.

private collectExternalCommits(externalRef, seedHash?): string[] [source:402]

Collects the list of commit hashes to replay, in chronological order (--reverse). If a seedHash is provided (set during setup), only collects commits after that point.

exported replayCommits(opts): { mirrored: number; upToDate: boolean } [source:426]

This is the heart of the system. It orchestrates the entire replay process:

  1. Build the set of already-synced commits (via buildAlreadySyncedSetFor)
  2. Find the seed hash (via findSeedHash)
  3. Collect all external commits that need replaying (via collectExternalCommits)
  4. Filter out already-synced ones
  5. For each new commit: generate patch → apply to subdirectory → stage files → commit with original metadata + sync trailer
  6. Handle empty patches (the commit exists in the external repo but produces no changes in our context) by recording an empty commit

Returns how many commits were mirrored and whether we were already up to date.

shadow-import.ts — Local Import Script

Purpose: Sync external changes and safely merge them into your local working branch. Runs ci-sync locally to fetch and replay external commits, then merges the shadow branch so only dir/ files are affected. No token needed.

How it works:

  1. Local sync — stashes the working tree, runs shadow-ci-sync.ts locally to fetch from external remotes and replay new commits into shadow branches, then restores the original branch and pops the stash. Skipped with --no-sync.
  2. Fetch the latest shadow branch from origin
  3. Check if up to date — if shadow is already an ancestor of HEAD, nothing to do
  4. Merge with --no-commit — merges the shadow branch but stops before committing, so we can fix the index
  5. Restore non-dir/ files — git read-tree HEAD resets the entire index to HEAD (undoing the merge's effect on all files in one command), then git checkout shadowRef -- dir/ brings in only the dir/ changes from the shadow branch. This is much more efficient than listing and checking out files in batches.
  6. Commit — the merge commit has correct ancestry and only dir/ files were changed

Why not a plain git merge?

The shadow branch's tree only contains dir/ files (the export strips everything else). A plain git merge origin/shadow/frontend/main would tell git that all non-dir/ files were "deleted" on the shadow side, causing git to delete your scripts, config, tests, and everything else outside dir/.

Always use npm run import, never raw git merge

This is critical. A raw git merge origin/shadow/{dir}/main will delete your files. The pull script exists specifically to prevent this. If you accidentally run a raw merge, restore with git checkout origin/main -- . and recommit.

shadow-export.ts — Local Export Script

Purpose: Export your local changes from an internal repo subdirectory to the shadow branch using git plumbing commands, filtering out files you don't want in the external repo. Syncs external changes first to ensure the shadow branch is up to date. No worktree or merge is needed for the export itself — the tree and commit are built directly in memory.

Step-by-step flow:

  1. Parse arguments — message (optional), remote, dir, branch, dry-run, no-sync
  2. Check for uncommitted changes in dir/ — refuses to run if the subdirectory has unsaved work. Acquire lock (acquireLock) to prevent concurrent exports.
  3. Parse .shadowignore (parseShadowIgnore) — load exclusion patterns
  4. Local sync — runs shadow-ci-sync.ts locally (stash, sync, restore) to ensure the shadow branch has the latest external commits. Skipped with --no-sync.
  5. Fetch from origin — verify the shadow branch exists and is merged into HEAD (i.e. git merge-base --is-ancestor shadowRef HEAD). If the shadow branch has unmerged commits, export refuses and tells you to import first.
  6. Set GIT_INDEX_FILE — point to a temp file so the real index is untouched
  7. Build the desired tree — git read-tree --empty clears the temp index, then git read-tree --prefix=dir/ HEAD:dir reads the subdirectory into the index. Also reads .github/ and shadow/ subtrees for CI dependencies.
  8. Remove shadowignored files from the index via git rm --cached
  9. git write-tree — creates a tree object hash from the index
  10. Compare tree against shadow branch's tree — if git rev-parse shadowRef^{tree} matches, exit with "no changes"
  11. Show diff with git diff-tree --stat so the user sees what will be exported
  12. Auto-generate commit message if none provided — lists commit subjects since the last export (e.g. Export backend/ (2 commits)\n- Add logger\n- Add config)
  13. git commit-tree with two parents (shadow tip + HEAD) — creates a proper merge commit without an actual merge operation. Adds a Shadow-export: true trailer so the forward workflow knows to forward this commit (ci-sync commits don't have this trailer and are skipped).
  14. Push to origin — if rejected (non-fast-forward), the user must merge the shadow branch and retry
  15. Cleanup — delete the temp index file

Why a merge commit instead of a snapshot?

The export creates a proper merge commit with two parents (shadow tip + HEAD) using git commit-tree. No worktree or actual git merge is involved — the tree is built entirely with git plumbing (read-tree + write-tree) and the commit is created directly. This means:

  • Real merge history — git log --graph shows the relationship between your branch and the shadow branch
  • Incremental diffs — each export commit shows only what actually changed, not a full replacement
  • No disk I/O — the entire operation happens in memory via a temp index file, making it faster and avoiding worktree setup

How .shadowignore stays hidden from the external repo

The export builds the tree using a temporary index (via GIT_INDEX_FILE), so only the files we explicitly read into it are included. Shadowignored files are removed from the index before git write-tree:

  1. git read-tree --prefix=dir/ HEAD:dir reads only the subdirectory into a clean temp index
  2. Shadowignored files are removed from the index via git rm --cached
  3. git write-tree creates the tree — it never contains the ignored files

The merge commit does point to HEAD as a parent, but the external repo never receives HEAD's ref — it only gets the shadow branch. So the ignored files are never reachable from the external side.

shadow-setup.ts — Bootstrap Script

Purpose: One-time initialization for each remote. Run once when you first start syncing with an external repo.

What it does:

  1. Fetches from the external remote
  2. Runs pre-flight checks (shallow clone, submodules, etc.)
  3. Records a seed commit on your current branch with the trailer Shadow-seed: backend a1b2c3d, telling CI sync "start after this commit." Without it, the first CI sync would try to replay the entire history of the external repo from the very beginning.

The shadow branch itself is created automatically by CI sync on its next run — setup only needs to record where to start.

shadow-ci-sync.ts — CI Pull Script

Purpose: Runs in GitHub Actions. Fetches from all external remotes and replays new commits into shadow branches.

What it does:

  1. Iterates over all configured remotes
  2. Resolves the URL from shadow-config.json (or env var override)
  3. Adds/updates the git remote
  4. Fetches from the external remote
  5. Lists all branches on the remote
  6. For each branch, checks out the corresponding shadow branch and calls replayCommits()
  7. Pushes updated shadow branches to origin
  8. Returns to detached HEAD between branches to avoid conflicts

Error handling: If one branch fails, it logs the error, resets any partial state, and continues with the next branch. Only fails at the end if any syncs failed.

New branches: If a new branch appears on the external remote, CI sync automatically discovers it (via listExternalBranches) and creates a new shadow branch for it.

Designed for CI, but safe to run locally via npm run import

This script uses git checkout to switch between shadow branches in the working tree. In CI this is safe because the runner starts clean. When run locally via shadow-import.ts, the import script handles stashing the working tree beforehand and restoring the original branch + working tree afterward. Do not run shadow-ci-sync.ts directly — use npm run import which wraps it safely.

shadow-ci-forward.ts — CI Push Script

Purpose: Runs in GitHub Actions when a shadow branch is pushed. Takes the content from the shadow branch and pushes it to the external remote. Uses git plumbing (no worktree, no archive, no tar).

What it does:

  1. Parses the branch name from GITHUB_REF_NAME (e.g. shadow/backend/main → dir=backend, branch=main). Finds the matching remote config entry.
  2. Sets GIT_INDEX_FILE to a temp file
  3. git read-tree origin/refName:dir — reads the dir/ subtree at root level (strips the prefix, so backend/src/app.ts becomes src/app.ts in the index)
  4. git write-tree — creates a tree hash from the index
  5. Compares tree against the external branch's tree — exits if no changes
  6. Builds the commit message: strips Shadow-* trailers from the shadow branch's message and adds a Shadow-forwarded-from: {hash} trailer so CI sync recognizes it as a round-trip commit and skips it on pull-back
  7. git commit-tree with the external branch tip as parent (or no parent for an orphan branch)
  8. Pushes to the external remote (requires EXTERNAL_REPO_TOKEN PAT)
  9. Cleanup: deletes the temp index file

Key details:

  • No filtering happens here — .shadowignore was already applied during export
  • The Shadow-forwarded-from trailer prevents round-trip duplicates: when CI sync pulls from external, it skips commits that originated from us
  • The export preserves CI dependencies on the shadow branch (.github/workflows/, package.json, shadow/shadow-ci-forward.ts, shadow/shadow-common.ts, shadow/shadow-config.json) so the forward workflow can run

9. Design Decisions & Why

Why per-commit replay instead of a single squashed diff?

Replaying commits one-by-one preserves the full history: who made each change, when, and why. A squashed diff would lose all intermediate commits and show everything as one big change by "sync bot." Per-commit replay means git blame and git log remain useful.

Why trailers instead of a database or state file?

Trailers are embedded in commit messages, so they travel with the code. They work in CI, in clones, and across branches without any external state to manage. There's no database to corrupt, no file to lose, and no sync state to get out of date. The downside is that scanning the log gets slower as history grows, but in practice the seed hash limits how far back it needs to look.

Why shadow branches instead of direct push/pull?

The external repo has files at root level (src/app.ts), but our internal repo has them in a subdirectory (backend/src/app.ts). Shadow branches act as a translation layer. They also decouple the timing: CI can sync on a schedule without affecting your working branch.

Why git plumbing for export/forward?

Export and ci-forward no longer use worktrees. They use git plumbing commands (read-tree + write-tree + commit-tree) with a temporary index file to build the tree and commit directly in memory. This is faster than worktrees (no disk I/O, no checkout, no cleanup) and avoids the complexity of managing a second working directory. CI sync still uses worktrees/checkout because it replays commits one-by-one and needs a working tree for git apply.

Why cross-platform git config overrides?

Windows, macOS, and Linux handle line endings (\r\n vs \n), file permissions, and Unicode differently. The overrides (core.autocrlf=false, core.safecrlf=false, core.precomposeunicode=true, core.quotepath=false) ensure patches apply identically regardless of which OS creates or applies them.

Why a lockfile?

If two exports run simultaneously, they'd both modify the shadow branch and one would overwrite the other. The lockfile uses PID checking to detect stale locks (from crashed processes) and atomic file creation (wx flag) to prevent race conditions.

Why no push retries?

Export no longer retries on push failure. If someone else pushes to the shadow branch between our fetch and push, git rejects the push as "non-fast-forward." The user must merge the shadow branch (npm run import) and re-run the export. This simplifies the code significantly, and the race condition is rare in practice since the shadow branch is only written by export and CI sync.

Why merge-based export instead of snapshot?

The export still creates a proper merge commit with two parents (shadow tip + HEAD), but now uses git commit-tree instead of git merge. The merge ancestry is preserved without needing an actual merge operation — the tree is built with plumbing commands and the commit simply records both parents. This produces clean incremental diffs, lets git handle renames and deletions naturally, and the commit tree is always clean (only dir/ files minus .shadowignore matches).

12. Troubleshooting

Someone pushed directly to a shadow/** branch

Shadow branches should only be written to by shadow-export and shadow-ci-sync. Direct pushes create commits without the expected trailers (Shadow-export:, Shadow-synced-from:), which breaks the sync cycle.

Symptoms

  • shadow-export refuses with "shadow has commits not merged into your local branch"
  • Unexpected files appear after shadow-import
  • ci-sync creates duplicate commits (the rogue commit has no Shadow-synced-from: trailer, so it doesn't get deduplicated)

What won't happen

  • No infinite loop — the forward workflow only triggers for commits with the Shadow-export: trailer, so the rogue commit won't be forwarded to the external repo
  • No data loss — your local branch and external repos are unaffected until you explicitly import or export

Fix — rogue commit was NOT yet imported into local

  1. Find the last valid commit on the shadow branch: 
    git fetch origin
git log origin/shadow/backend/main --oneline
# Look for the last commit with a Shadow-synced-from: or Shadow-export: trailer
  2. Force-push the shadow branch back: 
    git push origin <last-valid-hash>:refs/heads/shadow/backend/main --force

Fix — rogue commit WAS already imported into local

  1. Revert the import merge on your local branch: 
    git revert <merge-commit-hash>
git push origin main
  2. Then fix the shadow branch as above: 
    git push origin <last-valid-hash>:refs/heads/shadow/backend/main --force

Prevention

Never push directly to shadow/** branches. Use npm run export to push local changes and let CI sync handle external changes. Optionally, set up a GitHub branch ruleset on shadow/** to restrict pushes to specific users or the GitHub Actions bot.

13. Setting Up Cross-Repo Push Access (PAT)

Shadow Sync needs one fine-grained PAT for the CI forward workflow to push to external repos (the default GITHUB_TOKEN can only push to the repo the workflow runs in).

No local token is needed — npm run import runs ci-sync locally using your git remotes, which already have access.

CI Forward Token (pushes to external repos)

Create the token:

  1. Go to GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens
  2. Click Generate new token
  3. Fill in:
    FieldValue
    Token nameshadow-sync-forward
    ExpirationYour choice (90 days, 1 year, etc.)
    Resource ownerYour GitHub account or organization
    Repository accessOnly select repositories → select the external repos only (e.g. test-frontend, test-backend)
  4. Under Permissions → Repository permissions, set:
    PermissionLevelWhy
    ContentsRead and writePush commits to external repos
    Leave everything else at "No access."
  5. Click Generate token and copy it.

Add as repository secret:

  1. Go to your internal repo (e.g. github.com/you/test-main) → Settings → Secrets and variables → Actions
  2. Click New repository secret
  3. Name: EXTERNAL_REPO_TOKEN, Value: paste the token
  4. Click Add secret

The forward workflow is already configured to use this secret.

Fine-grained tokens expire. When your token expires, the forward workflow will start failing. Generate a new token and update the EXTERNAL_REPO_TOKEN secret. Consider setting a calendar reminder for a few days before expiration.

Verify It Works

  1. Run npm run import -- -r frontend — should show "Running local sync..." and fetch external changes
  2. Make a change under frontend/, commit, run npm run export -- -r frontend
  3. Check the Actions tab — the Shadow Forward workflow should trigger and push to the external repo
  4. Verify the change appears on the external repo

Why not use a classic PAT?

Classic PATs grant access to all repos in your account. Fine-grained tokens let you scope access to only the specific external repos that need it. This follows the principle of least privilege — if the token leaks, the blast radius is limited to those repos.

14. Git Commands Reference

Shadow Sync relies on a mix of git plumbing (low-level) and porcelain (high-level) commands. This section explains every git command used by the system.

Plumbing Commands (Low-Level)

These commands operate on git's internal objects (trees, commits, blobs) and are used by export and ci-forward to build trees and commits without a working directory.

git read-tree --empty

Clears the index completely. Used with GIT_INDEX_FILE to start with a blank slate in a temporary index file.

git read-tree --prefix=dir/ TREE

Reads a tree object into the index under a prefix. For example, git read-tree --prefix=backend/ HEAD:backend puts the backend/ subtree into the index with its prefix preserved. Used by export to selectively include only the wanted subtrees.

git write-tree

Creates a tree object from the current index and returns its hash. The tree is stored in git's object database. Used after building the desired index to produce the tree for the commit.

git commit-tree TREE -p PARENT -m MSG

Creates a commit object from a tree hash with specified parent commits. Multiple -p flags create a merge commit. Returns the new commit hash. Used by export (two parents: shadow tip + HEAD) and ci-forward (one parent: external branch tip, or none for orphan).

git rev-parse REF^{tree}

Resolves a ref (branch, commit hash) to its tree object hash. Used to compare the newly built tree against the shadow branch's existing tree to detect whether anything actually changed.

git diff-tree --stat TREE1 TREE2

Shows a summary of what changed between two trees (file names and line counts). Used by export to display what will be pushed before committing.

git ls-tree -r --name-only REF -- PATH

Lists all files in a tree at a specific path, recursively. Used by pre-flight checks to scan for submodules, symlinks, and case conflicts.

git symbolic-ref --short HEAD

Returns the current branch name (e.g. main) without the refs/heads/ prefix. Used by getCurrentBranch().

git rev-parse --verify REF

Checks if a ref exists and returns its hash. Exits with non-zero if the ref doesn't exist. Used by refExists().

git interpret-trailers --trailer KEY:VALUE

Appends a trailer (metadata line) to a commit message with proper formatting. Used by appendTrailer() to add Shadow-synced-from and Shadow-forwarded-from trailers.

Porcelain Commands (High-Level)

These commands are the familiar git operations that work with the working directory and index.

git fetch REMOTE

Downloads objects and refs from a remote repository. Used before every sync operation to ensure we have the latest state.

git push REMOTE SRC:DST

Pushes a local ref to a remote. Used by export (pushes to shadow branch), ci-forward (pushes to external remote), and ci-sync (pushes updated shadow branches).

git checkout REF -- PATH

Restores specific files from a ref into the working tree and index. Used by pull to bring in only dir/ changes from the shadow branch after resetting the index.

git apply --directory=DIR PATCH

Applies a patch under a subdirectory prefix. Used by the replay engine to apply external commits into the correct subdirectory (e.g. backend/).

git archive REF PATH

Creates a tar archive of files at a specific path from a ref. Documented here for reference, though no longer used by ci-forward (replaced by read-tree).

git merge-base --is-ancestor A B

Checks if commit A is an ancestor of commit B (exit code 0 = yes, 1 = no). Used by export to verify the shadow branch has been merged into HEAD before exporting.

git log --format=FORMAT --grep=PATTERN

Searches commit messages with custom output formatting. Used to find seed commits (by trailer) and to list commits since the last export for auto-generated messages.

git diff --cached --quiet

Checks if there are staged changes. Exit code 1 means yes (changes exist), 0 means no. Used to detect uncommitted work before export.

git ls-files

Lists files tracked in the index. Used with the temp index to enumerate files for shadowignore filtering.

git rm --cached

Removes files from the index without touching the working tree. Used by export to strip shadowignored files from the temp index before write-tree.

15. Security Considerations

Input validation

validateName() rejects remote/directory names that could be used for path traversal (..), absolute path injection (/etc/passwd), or git flag injection (--exec=...). This is critical because these names are interpolated into git commands and file paths.

.shadowignore

Prevents sensitive files (AI prompts, local configs, IDE settings) from leaking to external repos. Applied during export by removing matched files from the git index before committing. The resulting commit tree never contains ignored files, and since the external repo only receives the shadow branch (not your working branch), the ignored files are never reachable from the external side.

Committed content only

The export script merges committed state (HEAD), not the working directory. It also refuses to run if there are uncommitted changes in the target directory, preventing partial or broken state from being exported.
